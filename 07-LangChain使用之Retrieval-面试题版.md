## 2. 必会基础题

### 面试题1：什么是 Retrieval？它在 LangChain 中负责什么？

#### 标准回答

Retrieval 直译就是“检索”。在 LangChain 中，Retrieval 模块负责把外部知识接入到大模型应用中，通常覆盖数据加载、文档切分、向量化、向量存储、相似度检索、检索器封装等步骤。  
它的目标是让系统先从知识库中找出相关内容，再把这些内容提供给大模型生成答案。

#### 面试官想听到的关键词

- 检索
- 外部知识接入
- 文档处理流水线
- 检索增强生成
- RAG

#### 高分补充

如果只说“Retrieval 就是搜索文档”会显得太浅。更准确地说，Retrieval 是 LangChain 中面向 RAG 的基础能力集合，它不是单点功能，而是一整条工程链路。

#### 延伸追问

- Retrieval 和 RAG 是什么关系？
- Retrieval 模块通常包含哪些环节？

---

### 面试题2：为什么需要 Retrieval？它主要解决什么问题？

#### 标准回答

需要 Retrieval 的核心原因是大模型存在幻觉问题，并且模型参数中不可能稳定包含所有专业领域、私有领域和最新领域知识。  
Retrieval 通过在回答前从外部知识库中检索相关内容，再把检索结果作为上下文提供给大模型，从而提升回答的准确性、可控性和时效性。

#### 高分补充

它本质上是在解决两个问题：

- 模型知识不全
- 模型知识不够可靠

尤其在企业知识库、制度文档、技术手册、内部 SOP、客服问答这些场景下，Retrieval 比单纯依赖模型参数记忆更实用。

#### 延伸追问

- 为什么不直接微调模型？
- Retrieval 和 Prompt Engineering 相比有什么优势？

---

### 面试题3：什么是 RAG？

#### 标准回答

RAG 是 `Retrieval-Augmented Generation` 的缩写，即“检索增强生成”。它的基本流程是：先根据用户问题从知识库中检索相关文档，再把检索结果作为上下文输入给大模型，最后由大模型基于这些外部知识生成回答。

#### 高分补充

RAG 的本质不是让模型变聪明，而是让模型在生成前获得更可靠的证据。  
所以它通常适合知识密集型任务，而不是单纯的开放式聊天。

#### 延伸追问

- RAG 和微调的区别是什么？
- RAG 的优缺点分别是什么？

---

### 面试题4：RAG 的优缺点是什么？

#### 标准回答

RAG 的优点是：

- 能引入外部知识，缓解幻觉
- 知识更新灵活，不必重新训练模型
- 更适合私有知识和专业知识场景
- 比单纯 Prompt 提示更稳定

RAG 的缺点是：

- 链路更长，响应延迟更高
- 需要额外的嵌入、存储、检索成本
- 最终效果高度依赖切块质量、嵌入质量和检索质量
- 检索失败时，生成结果也会受到影响

#### 高分补充

RAG 不会从根上消灭幻觉，它只是用“外部知识约束”来降低幻觉概率。  
所以不能把 RAG 理解成绝对正确的保证机制。

## 3. 流程理解题

### 面试题5：请你描述一下一个典型 Retrieval / RAG 系统的完整流程。

#### 标准回答

一个典型流程可以概括为：

1. 准备数据源
2. 使用 Document Loader 将数据读取为 Document 对象
3. 使用 Text Splitter 将长文档切分成多个 chunk
4. 使用 Embedding 模型将 chunk 向量化
5. 将向量和文档写入 Vector Store
6. 用户提问时，将 query 也做向量化
7. 在向量库中检索最相关的 chunk
8. 将检索结果拼入 Prompt
9. 调用 LLM 基于上下文生成最终答案

#### 高分补充

面试时最好补一句：

`RAG 的效果核心取决于“切得对不对、召回准不准、Prompt 约束得稳不稳”。`

这句话很容易体现你不只是会背流程，而是理解关键瓶颈。

---

### 面试题6：Source、Load、Transform、Embed、Store、Retrieve 分别是什么意思？

#### 标准回答

- `Source`：原始数据源，比如 PDF、TXT、CSV、网页、JSON、代码、API 数据等
- `Load`：把原始数据加载为统一的 `Document` 对象
- `Transform`：对文档进行处理，比如切分、去重、提取元数据等
- `Embed`：把文本转换为向量表示
- `Store`：把文档向量和相关信息存到向量数据库
- `Retrieve`：根据用户问题从向量库里召回最相关的文档块

#### 高分补充

这里最容易被忽略的是 `Transform`。很多人会背流程，但不会强调“切块和元数据设计对效果影响非常大”。

---

## 4. Document 与 Loader 题

### 面试题7：LangChain 中的 Document 是什么？

#### 标准回答

Document 是 LangChain 中用于表示文档内容的统一数据结构，通常至少包含两个核心属性：

- `page_content`：正文内容
- `metadata`：元数据

无论原始数据来自 txt、pdf、csv、json 还是网页，经过 Loader 处理后通常都会转成 Document 对象，以便统一进入后续切分、向量化和检索流程。

#### 高分补充

可以把 Document 理解成 LangChain 在非结构化知识处理场景中的通用载体。  
它是 Loader、Splitter、VectorStore、Retriever 之间衔接的关键中间对象。

---

### 面试题8：为什么不同的 Loader 最终都能用统一方式处理？

#### 标准回答

因为 LangChain 对 Loader 做了统一抽象。不同的文档加载器虽然面向不同数据源，但最终都遵循类似的接口规范，通常通过 `load()` 返回 `Document` 列表。  
这体现了框架层的统一设计，使后续模块不需要关心原始数据具体来自哪里。

#### 高分补充

这其实是典型的“面向抽象编程”思想：上层只依赖统一接口，不依赖具体文件格式实现。

---

### 面试题9：常见的 Document Loader 有哪些？分别适合什么场景？

#### 标准回答

常见 Loader 包括：

- `TextLoader`：加载 txt 文本
- `PyPDFLoader`：加载 PDF
- `CSVLoader`：加载表格数据
- `JSONLoader`：加载 JSON，适合 API 返回和嵌套结构数据
- `UnstructuredHTMLLoader`：加载网页 HTML
- `UnstructuredMarkdownLoader`：加载 Markdown 文档
- `DirectoryLoader`：批量加载目录中的文件

#### 高分补充

如果面试官追问最容易踩坑的 Loader，我会优先说 `JSONLoader`，因为它往往需要用 `jq_schema` 精准抽取目标字段，而不是整个 JSON 原样塞进 page_content。

---

### 面试题10：JSONLoader 使用时最需要注意什么？

#### 标准回答

最需要注意的是：

- 只提取真正有价值的文本字段
- 合理区分哪些内容进 `page_content`
- 哪些字段保留在 `metadata`

通常需要配合 `jq_schema`、`content_key` 等参数，从嵌套 JSON 中精准提取目标文本。

#### 高分补充

做 RAG 时，`page_content` 应该尽量放“用于语义检索的正文”，`metadata` 则适合放来源、作者、时间、分类、页码等结构化信息。  
如果这个边界处理不好，召回效果通常会变差。

---

## 5. 文档切分题

### 面试题11：为什么 RAG 中通常必须做文档切分？

#### 标准回答

因为原始文档往往太长，直接整篇入库或整篇喂给模型会有两个主要问题：

- 包含大量无关信息，降低检索和生成质量
- 超出模型上下文窗口限制

所以通常要把长文档切成较小的 chunk，以 chunk 为单位进行向量化、存储和检索。

#### 高分补充

切块的本质是在平衡：

- 语义完整性
- 检索粒度
- Token 成本

这不是一个纯字符串处理问题，而是影响 RAG 效果的关键工程问题。

---

### 面试题12：chunk_size 和 chunk_overlap 分别起什么作用？

#### 标准回答

- `chunk_size` 表示每个文本块的最大长度
- `chunk_overlap` 表示相邻文本块之间重叠的内容长度

设置 `chunk_overlap` 的目的是避免重要信息刚好落在块边界，导致上下文断裂或召回缺失。

#### 高分补充

通常 overlap 不是越大越好。  
太小会丢边界信息，太大会造成冗余、增加存储和召回成本。实际中常结合业务文本特点来调优。

---

### 面试题13：CharacterTextSplitter 和 RecursiveCharacterTextSplitter 有什么区别？

#### 标准回答

`CharacterTextSplitter` 更偏简单的字符级切分，通常根据给定分隔符和长度做切块；  
`RecursiveCharacterTextSplitter` 会按多级分隔符递归尝试切分，比如先按段落、再按换行、再按空格、最后按字符级切分，因此更能保持自然语言边界和语义完整性。

#### 高分补充

真实项目中，`RecursiveCharacterTextSplitter` 往往是更常见、更推荐的默认选择，因为它比纯字符切分更灵活。

---

### 面试题14：为什么说切块策略会显著影响 RAG 效果？

#### 标准回答

因为检索是按 chunk 进行的。如果 chunk 切得过大，会混入太多噪声；切得过小，又会破坏语义完整性。  
切块策略直接影响：

- 文档向量质量
- 检索命中率
- 上下文完整性
- 最终生成答案的准确性

#### 高分补充

很多 RAG 项目的问题并不是模型不行，而是切块策略不合理。  
所以切块往往是最值得优先调优的环节之一。

---

### 面试题15：什么是 TokenTextSplitter？为什么按 token 切分更合理？

#### 标准回答

TokenTextSplitter 是按 token 数量而不是按字符数量进行切分的文本分割器。  
由于大模型的上下文窗口和计费机制通常都基于 token，因此按 token 切分能更准确地控制输入长度和成本。

#### 高分补充

要特别注意：

`字符数不等于 token 数`

尤其在中文场景下，字符和 token 的映射关系并不直观，所以按字符切可能看起来没超长，但实际上已经接近模型限制。

---

### 面试题16：什么是 SemanticChunker？它和传统切分有什么区别？

#### 标准回答

SemanticChunker 是一种基于语义边界的文本切分方式。它通常结合嵌入模型，根据相邻文本片段之间的语义差异来决定分块位置，而不是仅依赖字符数或分隔符。

相比传统切分：

- 优点是语义更完整
- 缺点是计算成本更高、速度更慢、实现更复杂

#### 高分补充

如果面试官问你什么时候会考虑 SemanticChunker，可以回答：

当普通切块导致语义断裂、召回质量不稳定，且系统愿意承担更高计算成本时，可以尝试语义切块。

---

## 6. Embedding 题

### 面试题17：什么是 Embedding？在 RAG 中起什么作用？

#### 标准回答

Embedding 是把文本转换成向量表示的过程。  
在 RAG 中，它的作用是把文档块和用户问题映射到同一个向量空间中，从而通过向量相似度实现语义检索。

#### 高分补充

Embedding 不是用来“生成答案”的，而是用来“支持检索”的。  
很多初学者会把 Embedding 模型和 LLM 混淆，这在面试里是减分点。

---

### 面试题18：embed_query 和 embed_documents 有什么区别？

#### 标准回答

- `embed_query`：用于对用户查询进行向量化
- `embed_documents`：用于对文档列表进行向量化

它们职责不同，但都会服务于同一个向量检索流程。

#### 高分补充

有些嵌入模型会对查询和文档使用不同的优化方式，所以接口层面通常会区分开。

---

### 面试题19：为什么说 Embedding 是向量检索的基础？

#### 标准回答

因为没有向量化，文本之间只能做关键词级匹配；而有了 Embedding，系统才能在向量空间里比较语义相似度，实现“意思接近”的检索，而不仅仅是“字面一致”的检索。

#### 高分补充

这也是 RAG 比传统全文检索更强的关键之一：它支持语义层面的召回。

---

## 7. Vector Store 题

### 面试题20：什么是 Vector Store？它和传统数据库有什么不同？

#### 标准回答

Vector Store 是专门用于存储文本向量并支持相似度检索的存储系统。  
它和传统数据库的区别在于，传统数据库擅长精确匹配和结构化查询，而向量数据库擅长高维向量的近似最近邻搜索与语义相似检索。

#### 高分补充

可以说传统数据库回答的是“是否完全匹配”，向量库回答的是“谁最相似”。

---

### 面试题21：常见的向量数据库有哪些？

#### 标准回答

LangChain 常见对接的向量库有：

- `FAISS`
- `Chroma`
- `Milvus`
- `Pinecone`
- `Redis` 等

#### 高分补充

如果是本地实验和教学演示，`FAISS` 和 `Chroma` 最常见；  
如果走服务化、分布式或生产环境，常会考虑 `Milvus`、`Pinecone` 等方案。

---

### 面试题22：Chroma 有哪些常见使用方式？

#### 标准回答

Chroma 常见有两种模式：

- 内存模式
- 持久化模式

如果不指定 `persist_directory`，通常是内存模式；  
如果指定持久化目录，则可以把索引落到本地磁盘，便于复用。

#### 高分补充

面试里可以顺带说明：  
Chroma 比较适合快速原型和本地实验，是很多 RAG 入门项目的常见选择。

---

### 面试题23：similarity_search、similarity_search_with_score、MMR 分别是什么？

#### 标准回答

- `similarity_search`：按相似度返回最相关文档
- `similarity_search_with_score`：返回文档及对应分数
- `max_marginal_relevance_search`：在相关性基础上兼顾多样性，减少重复内容

#### 高分补充

MMR 很适合文档内容重复度较高的知识库，因为它能避免前几个结果都非常相似，提升上下文覆盖面。

---

## 8. Retriever 题

### 面试题24：Retriever 和 Vector Store 的区别是什么？

#### 标准回答

Vector Store 更偏底层，负责向量的存储和相似度搜索；  
Retriever 更偏上层，负责按某种召回策略对外返回相关文档。

简单说：

- Vector Store 负责“存”和“搜”
- Retriever 负责“按策略召回”

#### 高分补充

在 LangChain 中，Retriever 是更统一的调用接口，便于后续和链、Agent 等模块集成。

---

### 面试题25：如何从向量库得到 Retriever？

#### 标准回答

通常通过：

`db.as_retriever()`

来把向量库包装成 Retriever。

之后就可以使用 `invoke()` 等方式执行检索。

#### 高分补充

`as_retriever()` 还可以传不同的 `search_type` 和 `search_kwargs` 来控制检索策略。

---

### 面试题26：Retriever 常见检索策略有哪些？

#### 标准回答

常见策略包括：

- 默认相似性搜索
- `similarity_score_threshold`
  只有分数高于阈值才返回

- `mmr`
  平衡相关性和多样性

#### 高分补充

如果知识库噪声较多，可以考虑阈值检索；  
如果知识库内容重复严重，可以考虑 MMR。

---

## 9. RAG 与 LLM 结合题

### 面试题27：Retriever 和 LLM 通常如何配合？

#### 标准回答

通常做法是：

1. 先使用 Retriever 检索出相关文档块
2. 把这些文档块拼接到 Prompt 的上下文部分
3. 再调用 LLM 基于这些上下文生成答案

这就是最典型的 RAG 实现方式。

#### 高分补充

为了降低幻觉，Prompt 往往会加入明确约束，比如：

- 仅依据给定上下文回答
- 如果上下文中没有答案，就直接说明不知道
- 不要编造信息

---

### 面试题28：为什么即使大模型本身能回答某些问题，仍然需要 RAG？

#### 标准回答

因为模型自己的参数知识可能过时、不完整，也不一定包含企业私有知识。  
对于公司内部文档、行业手册、最新资料等内容，仅依赖模型参数记忆是不可靠的，而 RAG 可以通过外部知识补充来提升可控性和准确性。

#### 高分补充

一句很适合面试的表达是：

`RAG 的价值不是替代模型常识，而是增强模型在特定知识场景下的可信回答能力。`

---

### 面试题29：RAG 和微调有什么区别？

#### 标准回答

RAG 是把知识放在模型外部，通过检索按需取用；  
微调是把知识学习到模型参数中。

两者区别主要体现在：

- 知识位置不同
- 更新成本不同
- 适用场景不同

#### 高分补充

通常：

- 需要频繁更新知识时，更偏向 RAG
- 需要稳定学习某类任务模式或输出风格时，才更可能考虑微调

---

## 10. 进阶理解题

### 面试题30：RAG 效果不好时，你会优先排查哪些环节？

#### 标准回答

我会优先排查：

1. 文档是否加载正确
2. 切块策略是否合理
3. Embedding 模型是否适合当前语料
4. 检索策略是否合适
5. Prompt 是否限制清楚
6. metadata 是否设计合理

#### 高分补充

如果只能说一个最重要的点，我会优先看：

`切块策略和召回结果`

因为很多 RAG 的问题不是生成模型不行，而是检索阶段就没把正确证据取回来。

---

### 面试题31：metadata 在 Retrieval 中为什么重要？

#### 标准回答

metadata 可以保存来源、页码、标题、作者、分类、时间等结构化信息。  
它的重要性在于：

- 可以支持过滤检索
- 可以支持结果溯源
- 可以帮助后续展示引用来源
- 可以参与更精细的检索策略设计

#### 高分补充

高质量 RAG 往往不只是“召回内容”，还要“说清内容来自哪里”。这时 metadata 就非常关键。

---

### 面试题32：为什么说 Retrieval 不只是一个“查数据库”的功能？

#### 标准回答

因为 Retrieval 并不是单纯执行一次查询，而是覆盖了从异构数据接入、文档标准化、文本切分、向量化、存储、召回到最终和 LLM 集成的整条链路。  
它本质上是一套面向知识增强的工程体系。

#### 高分补充

这个回答很适合体现你的系统观，而不是只停留在 API 使用层。

---

## 11. 面试中的高频追问模板

### 追问1：你项目里为什么选 RecursiveCharacterTextSplitter？

#### 参考回答

因为它在自然语言文本里通常能较好地兼顾语义完整性和实现复杂度。  
相比纯字符硬切，它会按多级分隔符递归尝试，更适合大多数普通文本场景，适合作为默认切块方案。

---

### 追问2：你项目里为什么选 Chroma 或 FAISS？

#### 参考回答

如果是本地实验、教学项目或快速原型，我会优先考虑 Chroma 或 FAISS，因为它们上手快、集成简单、适合本地验证 Retrieval 效果。  
如果是更大规模或服务化场景，再考虑 Milvus、Pinecone 这类方案。

---

### 追问3：你如何控制检索质量？

#### 参考回答

我通常会从四个方面控制：

- 优化切块策略
- 选择合适的 embedding 模型
- 调整检索参数，比如 `k`、阈值、MMR
- 在 Prompt 中加强基于上下文回答的约束

## 12. 一套适合初学者背诵的总答案

如果面试官让你整体介绍 Retrieval / RAG，你可以这样答：

`在 LangChain 中，Retrieval 是围绕检索增强生成构建的一整套能力，主要包括数据加载、文档切分、向量化、向量存储和检索器封装。它的目标是先从外部知识库中召回和问题相关的文档，再把这些内容作为上下文提供给大模型生成答案，从而缓解大模型幻觉问题，提高私有知识和专业知识场景下的回答准确性。一个典型 RAG 流程是：Source -> Load -> Split -> Embed -> Store -> Retrieve -> LLM Generate。其中，文档切分策略、Embedding 模型选择、向量检索策略和 Prompt 约束方式，都会显著影响最终效果。`

这段话如果你能顺下来，已经足够应对大多数基础面试问题。
