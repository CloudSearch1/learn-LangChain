## 2. 必会概念题

### 面试题1：什么是 LangChain？

#### 标准回答

LangChain 是一个面向大语言模型应用开发的开源框架，用来把提示词、模型调用、输出解析、检索、记忆、工具调用、Agent 等能力进行标准化和组件化封装，从而更高效地构建问答系统、RAG 系统、智能体、知识库助手等应用。

它本身不是大模型，而是围绕大模型构建应用的工程框架。

#### 面试官想听到的关键词

- 开源框架
- LLM 应用开发
- 组件化
- 标准化
- 不是模型本身

#### 延伸追问

- 它和 OpenAI SDK 有什么区别？
- 不用 LangChain 能不能开发？

#### 答题提醒

不要把 LangChain 说成“一个大模型”或者“一个知识库”。

---

### 面试题2：为什么需要 LangChain？直接调大模型 API 不行吗？

#### 标准回答

直接调用大模型 API 可以完成简单任务，但在真实业务中，往往还需要处理 Prompt 模板化、模型切换、结构化输出、文档检索、工具调用、多轮对话记忆、链路监控等问题。

LangChain 的价值在于：

- 提供统一抽象，降低多模型切换成本
- 提供链式编排能力，便于组织复杂流程
- 提供 Retrieval、Memory、Agent 等现成模块
- 便于从 Demo 过渡到工程化应用

所以不是“能不能不用”，而是“当场景复杂时，用框架更省事、更规范、更容易维护”。

#### 高分补充

如果需求只是单轮文本生成，LangChain 不一定是必需的；但只要进入 RAG、Agent、多步骤链路、生产级可观测阶段，LangChain 的优势会明显体现出来。

#### 延伸追问

- LangChain 解决了哪些原生 API 不方便解决的问题？
- 它最大的工程价值是什么？

---

### 面试题3：LangChain 和 LLM 是什么关系？

#### 标准回答

LLM 是负责生成和推理的基础模型，LangChain 是围绕 LLM 构建应用的框架。前者像“智能引擎”，后者像“应用装配平台”。

没有 LLM，LangChain 不能独立完成文本生成；没有 LangChain，也可以直接调用 LLM，但复杂应用的组织、扩展和维护成本会更高。

#### 好的表达方式

可以类比为：

- `LLM 负责能力`
- `LangChain 负责编排`

---

### 面试题4：LangChain 适合做哪些应用？

#### 标准回答

LangChain 适合构建以下类型的应用：

- 文档问答系统
- 企业知识库
- 智能客服
- 多轮对话机器人
- Agent 智能体系统
- 数据库问答
- 工具调用型 AI 助手

这些场景的共同特点是：不只是“调用一次模型”，而是要把模型、数据、工具和流程组合起来。

## 3. 架构与生态题

### 面试题5：LangChain 生态中除了 LangChain 本体，还有哪些重要模块？

#### 标准回答

主要有三个非常重要的扩展模块：

- `LangGraph`：用于构建更复杂的 Agent 工作流和状态流转
- `LangSmith`：用于调试、测试、监控、链路追踪和评估
- `LangServe`：用于把 LangChain 的链或 Runnable 部署为 REST API 服务

#### 深度回答

如果从工程阶段看：

- LangChain 负责“开发应用逻辑”
- LangGraph 负责“复杂流程编排”
- LangSmith 负责“调试观测和评估”
- LangServe 负责“服务化部署”

#### 延伸追问

- 为什么 LangSmith 很重要？
- LangGraph 和普通 Chain 的区别是什么？

---

### 面试题6：什么是 LangGraph？它和 LangChain 是什么关系？

#### 标准回答

LangGraph 可以理解为建立在 LangChain 基础上的复杂工作流框架，特别适合表达有状态、多步骤、分支、循环和多 Agent 协作的流程。

LangChain 更偏组件和基础链式开发，而 LangGraph 更偏复杂流程控制和状态管理。

#### 面试加分点

可以补充一句：

如果普通 Chain 更像“线性流程”，那么 LangGraph 更像“可编排的状态机或工作流图”。

---

### 面试题7：什么是 LangSmith？为什么说它在生产环境里很重要？

#### 标准回答

LangSmith 是 LangChain 生态中的调试、测试、评估和监控平台，可以帮助开发者查看每一步调用链路、Prompt、模型输入输出、工具调用过程以及运行结果。

它之所以重要，是因为 LLM 应用的失败经常不是传统报错，而是：

- 检索不到正确内容
- Prompt 拼接错误
- 工具选择错误
- 输出不符合预期

这些问题如果没有可观测平台，很难定位。

#### 高分表达

传统程序主要调代码，LLM 应用除了调代码，还要调 Prompt、调链路、调检索、调工具、调模型行为，所以可观测性在 AI 工程里比传统应用更重要。

---

### 面试题8：LangServe 有什么作用？

#### 标准回答

LangServe 用于把 LangChain 的链、Runnable 或应用能力包装成 REST API 对外暴露，便于前端、Java 服务、其他后端系统调用。

它的意义在于把本地 demo 变成可集成、可部署、可服务化的能力。

## 4. 技术路线与选型题

### 面试题9：大模型应用开发常见的四种方案是什么？

#### 标准回答

常见有四种：

- 纯 Prompt
- Agent + Function Calling
- RAG
- Fine-tuning 微调

#### 它们的定位分别是什么

- 纯 Prompt：适合简单文本生成和轻量交互
- Agent + Function Calling：适合需要调用外部工具和多步决策的任务
- RAG：适合需要引入外部知识、减少知识过时和幻觉的场景
- 微调：适合需要长期固化能力或风格，但成本最高

#### 追问：实际项目如何选择？

优先级通常是：

1. 先尝试纯 Prompt
2. 不够用再上 RAG
3. 如果需要外部行动能力，再考虑 Tool / Agent
4. 最后才考虑微调

---

### 面试题10：什么是 RAG？它解决了什么问题？

#### 标准回答

RAG 全称是 Retrieval-Augmented Generation，即检索增强生成。它的核心思想是先从外部知识库中检索与用户问题相关的信息，再将这些信息作为上下文交给大模型生成答案。

它主要解决两个问题：

- 大模型知识冻结
- 大模型幻觉

#### 更专业的回答

RAG 通过把“参数记忆”扩展为“外部知识检索 + 生成”模式，使系统能够利用最新、私有、垂直领域的数据回答问题，从而提高回答的准确性、时效性和可依据性。

#### 延伸追问

- RAG 为什么比单纯 Prompt 更适合企业知识库？
- RAG 的性能瓶颈通常在哪里？

---

### 面试题11：RAG 的完整流程是什么？

#### 标准回答

一个典型 RAG 流程包括：

1. 加载原始文档
2. 对文档做切分
3. 使用 Embedding 模型将文本块向量化
4. 将向量存入向量数据库
5. 用户提问
6. 将问题向量化
7. 做相似度检索
8. 取回相关文本片段
9. 把检索结果拼进 Prompt
10. 调用 LLM 生成最终回答

#### 高分加一句

在高精度场景下，通常还会加入 Reranker 对召回结果进行二次排序。

---

### 面试题12：RAG 中最关键的难点有哪些？

#### 标准回答

核心难点通常包括：

- 文档解析质量
- 文本切块策略
- 检索召回效果
- 重排序质量
- Prompt 组装方式

#### 为什么这题重要

因为很多人以为做 RAG 就是“换个模型 + 上个向量库”，但实际效果更多取决于检索链路而不是模型名字。

#### 延伸追问

- 为什么 chunk size 很关键？
- 什么时候需要 rerank？

---

### 面试题13：什么是 Reranker？什么时候要用？

#### 标准回答

Reranker 是对初步召回结果进行二次排序的模型或机制。第一阶段检索通常追求速度和召回率，Reranker 再对召回结果做更精细的相关性判断，以提升最终送给 LLM 的上下文质量。

适合使用的场景：

- 专业知识库
- 法律、医疗、金融等高精度场景
- 对答案相关性要求很高的客服系统

不适合的场景：

- 对时延要求极高的系统

因为它会增加额外计算成本。

---

### 面试题14：什么是 Agent？请给出一个较完整定义。

#### 标准回答

Agent 是一种基于大模型的智能体架构，不只是生成文本，还能结合记忆、工具和规划能力，分步骤完成更复杂的任务。

可以用一个公式概括：

`Agent = LLM + Memory + Tools + Planning + Action`

也就是说，Agent 的重点在于：

- 理解目标
- 拆解任务
- 选择工具
- 执行动作
- 根据反馈继续推进

#### 追问

- Agent 和普通聊天机器人有什么本质区别？
- Agent 为什么更难做稳定？

---

### 面试题15：Agent 的核心组成有哪些？

#### 标准回答

Agent 的核心组成通常包括五部分：

- `LLM`：负责理解、推理和决策
- `Memory`：负责保存上下文和经验
- `Tools`：负责调用外部能力
- `Planning`：负责任务分解和策略制定
- `Action`：负责真正执行动作

#### 高分理解

这五部分中，LLM 决定“怎么想”，Tool 决定“能做什么”，Planning 决定“先做什么后做什么”，Memory 决定“是否记得之前发生过什么”，Action 决定“是否真正落地执行”。

---

### 面试题16：Agent 和 RAG 的区别是什么？

#### 标准回答

RAG 主要解决“知识从哪里来”的问题，核心是检索外部资料再辅助生成；Agent 主要解决“任务怎么完成”的问题，核心是规划、调用工具和执行动作。

可以简单记成：

- `RAG 负责补知识`
- `Agent 负责做事情`

#### 再深入一点

两者不是对立关系，很多系统会结合使用。比如 Agent 可以把检索器当成一个工具，在需要时主动检索知识。

---

### 面试题17：什么时候该用 Agent，什么时候不该用？

#### 标准回答

当任务需要：

- 多步规划
- 动态决策
- 调用多个工具
- 根据中间结果调整后续行为

时，适合考虑 Agent。

如果任务只是：

- 简单问答
- 单次生成
- 固定流程处理

那么普通 Chain 或 RAG 通常更稳定、更低成本。

#### 高分表达

Agent 不是默认优选，而是复杂任务下的增强手段。它带来更强灵活性，也带来更高的不确定性、成本和调试难度。

---

## 5. 核心组件题

### 面试题18：LangChain 的核心组件有哪些？

#### 标准回答

LangChain 的核心组件通常包括六大模块：

- Model I/O
- Chains
- Memory
- Agents
- Retrieval
- Callbacks

#### 简要说明

- Model I/O：处理模型输入输出
- Chains：组织链式流程
- Memory：管理上下文
- Agents：负责自主决策和工具调用
- Retrieval：支持 RAG
- Callbacks：负责日志、监控、流式等钩子能力

---

### 面试题19：什么是 Model I/O？

#### 标准回答

Model I/O 是 LangChain 中用于标准化模型输入输出的模块，通常包括：

- Prompt Template
- Models
- Output Parser

它的职责是把原始输入组织成模型可处理的格式，调用模型生成结果，再把结果解析成业务需要的形式。

#### 延伸追问

- 为什么 Output Parser 很重要？
- Prompt Template 和普通字符串有什么区别？

---

### 面试题20：什么是 Prompt Template？为什么不用直接拼字符串？

#### 标准回答

Prompt Template 是对 Prompt 的模板化封装，支持变量填充、角色划分和统一管理。相比直接拼字符串，它更有利于复用、维护、调试和规范化。

在复杂系统里，Prompt 往往包含：

- system 指令
- user 输入
- 检索结果
- 输出格式要求

如果直接手工拼接，容易混乱且难维护。

#### 面试加分点

Prompt Template 的本质不是“省几行代码”，而是让 Prompt 从一次性文本变成可管理的工程资产。

---

### 面试题21：什么是 Output Parser？为什么它重要？

#### 标准回答

Output Parser 用于将模型输出解析成指定格式，例如字符串、JSON 或结构化对象。它的重要性在于，大模型天然擅长输出自然语言，但业务系统更需要可编程、可校验、可传递的结构化数据。

#### 举例

如果下游系统需要：

- `question`
- `answer`
- `confidence`

这样的字段，那么 Output Parser 就能帮助我们把模型输出规范成结构化格式。

---

### 面试题22：什么是 Chain？为什么说它是 LangChain 中最重要的模块之一？

#### 标准回答

Chain 是将多个处理步骤串联起来形成完整流程的机制，例如把 Prompt、LLM、Output Parser 连接成一个执行链。

它之所以重要，是因为真实 AI 应用通常包含多个步骤，而不是单次模型调用。Chain 让流程更清晰、更模块化、更容易复用和扩展。

#### 高分理解

Chain 的核心价值在于“把复杂处理流程表达成稳定可复用的管道”。

---

### 面试题23：常见的 Chain 类型有哪些？

#### 标准回答

常见类型包括：

- `LLMChain`：最基础的模型调用链
- `SequentialChain`：多个链顺序执行
- `RouterChain`：根据用户需求路由到不同链
- `RetrievalQA`：结合检索做问答

#### 答题提醒

如果面试不是特别细，不必背所有类名，但一定要知道“链的作用是什么”。

---

### 面试题24：什么是 Memory？它解决了什么问题？

#### 标准回答

Memory 是 LangChain 中用于保存对话历史和上下文的模块，主要解决多轮交互中“模型如何记住前文”的问题。

没有 Memory，模型每次调用都可能只看到当前输入；有了 Memory，系统可以把历史信息重新注入上下文，维持连续对话体验。

#### 延伸追问

- 短期记忆和长期记忆有什么区别？
- 为什么不能无限保存全部历史？

---

### 面试题25：常见的 Memory 类型有哪些？

#### 标准回答

常见的 Memory 类型有：

- `ConversationBufferMemory`：保存完整对话
- `ConversationSummaryMemory`：保存摘要
- `ConversationSummaryBufferMemory`：缓冲和摘要结合
- `VectorStoreRetrieverMemory`：把历史转为可检索记忆

#### 深度回答

不同记忆机制的核心差别在于如何平衡：

- 信息完整度
- token 成本
- 上下文长度限制

---

### 面试题26：什么是 Retrieval 模块？

#### 标准回答

Retrieval 模块是 LangChain 中支撑 RAG 的核心能力，负责把外部数据加载、切分、向量化、存储并检索出来，再提供给 LLM 作为上下文。

其典型链路包括：

- Source
- Load
- Transform
- Embed
- Store
- Retrieve

---

### 面试题27：什么是 Callbacks？在什么场景下有用？

#### 标准回答

Callbacks 是在 LangChain 应用运行各阶段插入钩子逻辑的机制，可用于日志记录、链路跟踪、流式输出、监控、性能分析等。

它的价值主要体现在工程化和生产化阶段，而不仅仅是 demo 开发。

---

## 6. 实践与代码理解题

### 面试题28：LangChain 中最简单的 Hello World 包含哪些步骤？

#### 标准回答

最简单的 Hello World 一般包括：

1. 配置环境变量
2. 创建模型实例，如 `ChatOpenAI`
3. 调用 `invoke`
4. 获取模型返回结果

如果稍微完整一点，还会加上：

5. Prompt Template
6. Output Parser

#### 面试官真正想考察的是

你是否知道 LangChain 最小闭环是怎么组成的，而不是只会背某段代码。

---

### 面试题29：为什么很多示例都要通过 `.env` 管理密钥？

#### 标准回答

因为 API Key、Base URL 等配置属于敏感信息或环境相关配置，不应该硬编码进代码。通过 `.env` 或系统环境变量管理，能提高安全性、可移植性和部署灵活性。

#### 面试加分点

在真实项目中，除了 `.env`，还会结合：

- 密钥管理系统
- CI/CD 环境变量
- 配置中心

---

### 面试题30：为什么 LangChain 示例里经常把 Prompt、LLM、Parser 串起来？

#### 标准回答

因为这三者正好构成了一个最典型的最小处理链：

- Prompt 负责输入组织
- LLM 负责生成
- Parser 负责输出结构化

这条链可以看作几乎所有更复杂应用的基础骨架。

---

### 面试题31：向量数据库在 RAG 中扮演什么角色？

#### 标准回答

向量数据库用于存储文档片段的向量表示，并在用户提问时根据问题向量进行相似度检索，从而找出语义相关的文本内容。

它的核心价值不是“存文本”，而是“支持语义相似搜索”。

#### 延伸追问

- 为什么不用传统关键词搜索？
- FAISS 是不是严格意义上的数据库？

---

### 面试题32：Embedding 模型在 RAG 中起什么作用？

#### 标准回答

Embedding 模型负责把文本映射成向量，使文本之间可以通过向量距离进行语义相似度计算。

在 RAG 中通常有两类向量化动作：

- 把知识库文档向量化
- 把用户问题向量化

然后通过相似度匹配完成检索。

---

### 面试题33：为什么说 RAG 的回答“更可依据”？

#### 标准回答

因为 RAG 会先检索外部资料，再基于这些资料生成答案。也就是说，回答不是只依赖模型参数中的记忆，而是依赖当前召回的上下文证据，因此更容易追溯依据，也更适合企业知识库等需要可信来源的场景。

---

### 面试题34：Agent 中 Tool 的作用是什么？

#### 标准回答

Tool 是 Agent 连接外部世界的手段。大模型本身只能生成文本，而 Tool 可以让 Agent 调用搜索、数据库、计算器、企业 API、检索器等外部能力，从而突破模型原生能力边界。

#### 高分补充

在很多系统里，检索器本身也可以封装成 Tool，所以 Agent 和 RAG 可以结合。

---

### 面试题35：什么是 AgentExecutor？

#### 标准回答

AgentExecutor 是用于运行 Agent 并协调工具调用执行过程的组件。它负责把 Agent 的决策转化为具体执行流程，包括调用哪个工具、接收工具结果、继续后续推理等。

---

## 7. 对比题

### 面试题36：Prompt、RAG、Agent、Fine-tuning 的差别怎么讲？

#### 标准回答

- `Prompt`：主要通过输入设计引导模型行为，不引入外部知识和复杂执行逻辑
- `RAG`：通过检索外部知识增强回答
- `Agent`：通过规划和工具调用完成复杂任务
- `Fine-tuning`：通过训练调整模型参数，固化风格或能力

#### 记忆口诀

- Prompt 调输入
- RAG 补知识
- Agent 做事情
- 微调改模型

---

### 面试题37：普通 Chain 和 Agent 的区别是什么？

#### 标准回答

普通 Chain 的流程大多是开发者预先定义好的，执行路径相对固定；Agent 则允许模型根据当前任务和上下文动态决定下一步做什么、是否调用工具、调用什么工具。

因此：

- Chain 更可控
- Agent 更灵活

但同时：

- Chain 更稳定
- Agent 更复杂

---

### 面试题38：Memory 和 RAG 有什么区别？

#### 标准回答

Memory 更强调会话历史和上下文连续性，通常服务于对话过程；RAG 更强调从外部知识库检索信息，通常服务于知识补充。

简单说：

- Memory 解决“刚刚聊过什么”
- RAG 解决“系统还知道什么”

---

## 8. 开放题与综合题

### 面试题39：如果让你设计一个企业知识库助手，你会怎么选型？

#### 参考回答

如果目标是企业文档问答，我会优先选择 RAG 方案，而不是一上来做 Agent 或微调。

基础架构会包括：

- 文档加载与清洗
- 文本切分
- Embedding
- 向量存储
- Retriever
- Prompt Template
- LLM
- Output Parser
- 日志与评测

如果后续需要更复杂的流程，比如自动调用企业系统、执行多步任务，再考虑引入 Agent。

#### 面试加分点

可以补充：

我会优先准备测试集和评测标准，而不是只看“感觉回答得像不像”。

---

### 面试题40：如果做一个“天气查询 + 行程建议”的 AI 助手，你会偏向 RAG 还是 Agent？

#### 参考回答

我会偏向 Agent，原因是这个任务不仅要回答，还要调用外部天气服务，并根据返回结果进行进一步推理和建议，属于典型的工具调用场景。

如果还需要参考固定旅游攻略或公司差旅制度，也可以在 Agent 中结合 RAG 检索相关资料。

---

### 面试题41：LangChain 学习顺序应该是什么？

#### 参考回答

我认为合理顺序是：

1. 先学 Model I/O
2. 再学 Retrieval / RAG
3. 再学 Chains
4. 再学 Memory
5. 最后学 Agents
6. 进阶再看 LangSmith 和 LangGraph

因为 Model I/O 是所有能力的基础，RAG 是最常见落地方向，Agent 则建立在前面能力都比较熟悉的前提上。

---

### 面试题42：LangChain 学习中最容易掉进哪些误区？

#### 参考回答

常见误区包括：

- 只背 API，不理解模块解决什么问题
- 只关注模型，不关注检索和数据质量
- 一上来就学 Agent，忽略基础链路
- 认为 RAG 效果不好一定是模型不行
- 忽视调试、评测和监控

#### 高分补充

尤其在 RAG 项目中，效果往往更多取决于文档处理和检索链路，而不是底层模型本身。

## 9. 高频一句话快答

### Q1：LangChain 是模型吗？

不是，它是围绕大模型开发应用的框架。

### Q2：RAG 的核心价值是什么？

把外部知识接入回答流程，减少知识过时和幻觉。

### Q3：Agent 的核心价值是什么？

让模型具备规划、工具调用和执行能力。

### Q4：LangSmith 是干什么的？

做调试、追踪、测试、评估和监控。

### Q5：为什么要 Output Parser？

因为业务系统需要结构化输出，而不是随意自然语言。

### Q6：为什么要 Memory？

为了让多轮对话保留上下文。

### Q7：为什么要向量数据库？

为了做语义相似检索，而不是只做关键词匹配。

### Q8：Agent 一定比 Chain 高级吗？

不一定，它更灵活，但也更难控、更贵、更复杂。

## 10. 面试收尾总结模板

如果面试官问你“你怎么理解 LangChain”，你可以用下面这段作为总结型回答：

LangChain 本质上是一个 LLM 应用开发框架，它的重点不是替代大模型，而是把 Prompt、模型调用、检索、记忆、工具和 Agent 这些能力标准化，帮助开发者更高效地搭建复杂 AI 应用。  
从应用落地角度看，我觉得最重要的两条主线是 RAG 和 Agent。RAG 解决知识问题，Agent 解决行动问题。  
从工程角度看，除了运行时能力，像 LangSmith 这样的调试监控工具也很关键，因为 LLM 应用的可观测性比传统系统更重要。
