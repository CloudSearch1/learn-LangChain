# 《02-LangChain使用之Model IO》面试题版

## 1. 使用说明

这份文档把《Model I/O》这一章整理成可面试表达的形式，重点不是背类名，而是帮你建立下面这种回答结构：

- 先说定义
- 再说作用
- 再说适用场景
- 再说区别和取舍

如果你能用这个结构讲出来，面试官会觉得你不只是“看过课”，而是真的理解了。

---

## 2. 核心总览题

### 面试题1：什么是 LangChain 的 Model I/O？

#### 标准回答

Model I/O 是 LangChain 中负责和大模型交互的核心模块，主要包括三个部分：

- `Prompt Template`：负责输入格式化
- `Model`：负责调用大模型
- `Output Parser`：负责输出解析

也就是说，Model I/O 对应的是大模型应用中最基础的一条链路：

`输入组织 -> 模型推理 -> 输出结构化`

#### 高分表达

Model I/O 不是单纯的“调一下模型”，而是把大模型交互做成标准化流程，这样上层应用才能更稳定、更可维护。

#### 面试官想听到的关键词

- 输入格式化
- 模型调用
- 输出解析
- 标准化
- 结构化

---

### 面试题2：为什么说 Model I/O 是 LangChain 里最基础也最重要的模块？

#### 标准回答

因为后面的 Chains、RAG、Memory、Agents，本质上都建立在和模型交互的基础之上。而 Model I/O 恰好定义了最底层的交互方式：

- 输入怎么组织
- 模型怎么调用
- 输出怎么变成下游程序能处理的数据

如果这一层没搞明白，后面做复杂链路时很容易只会写代码，不理解数据是怎么流动的。

---

## 3. 模型分类题

### 面试题3：LangChain 中的模型按功能通常分哪几类？

#### 标准回答

按功能通常分为三类：

- `LLMs / Text Models`：非对话模型
- `Chat Models`：对话模型
- `Embedding Models`：嵌入模型

#### 进一步说明

- 非对话模型主要处理字符串输入输出
- 对话模型处理消息列表，更适合现代大模型应用
- 嵌入模型负责把文本转成向量，主要用于检索和相似度计算

---

### 面试题4：什么是非对话模型 LLM？它的特点是什么？

#### 标准回答

非对话模型通常接受字符串作为输入，返回字符串作为输出，适合单次文本生成任务，比如摘要、翻译、单轮问答等。

它的特点是：

- 输入输出形式简单
- 不强调角色消息结构
- 不原生支持多轮对话上下文

#### 延伸追问

- 为什么现在更推荐 Chat Model？

---

### 面试题5：什么是 Chat Model？为什么它是现在的主流？

#### 标准回答

Chat Model 是基于消息结构进行交互的模型，输入通常是消息列表，每条消息带有角色信息，比如 `system`、`human`、`ai`，输出一般是 `AIMessage`。

它之所以成为主流，是因为现代大模型接口大多采用聊天式消息协议，这种方式更适合：

- 多轮对话
- 角色控制
- 上下文管理
- 工程化扩展

#### 高分表达

即使任务不是聊天，比如摘要或抽取，很多时候仍然优先使用 Chat Model，因为它的接口设计更通用。

---

### 面试题6：什么是 Embedding Model？它和 Chat Model 的区别是什么？

#### 标准回答

Embedding Model 的作用不是生成回答，而是把文本映射成向量，用于语义检索、相似度计算、聚类和 RAG 等场景。

区别在于：

- Chat Model 负责“生成”
- Embedding Model 负责“向量表示”

#### 一句话快答

Embedding 不是回答问题的模型，而是把文本转换成机器可比较的向量表示。

---

## 4. 参数与配置题

### 面试题7：调用大模型时常见的重要参数有哪些？

#### 标准回答

常见重要参数包括：

- `api_key`：身份认证
- `base_url`：模型服务地址
- `model / model_name`：具体模型名称
- `temperature`：控制随机性
- `max_tokens`：限制输出长度

#### 追问：哪些是最关键的？

最关键的是：

- 认证能不能通过
- 服务地址对不对
- 调的是哪个模型

而 `temperature` 和 `max_tokens` 更偏生成效果和成本控制。

---

### 面试题8：`temperature` 是什么？它越低是不是代表模型越聪明？

#### 标准回答

`temperature` 用于控制生成结果的随机性和多样性。值越低，输出通常越稳定、越保守；值越高，输出通常越发散、越有创意。

它不代表模型是否更聪明，而是代表采样风格更稳定还是更多样。

#### 适用建议

- 事实问答、抽取、分类：温度低一些
- 创意写作、文案生成：温度可以高一些

---

### 面试题9：`max_tokens` 的作用是什么？

#### 标准回答

`max_tokens` 用于限制模型输出的最大 token 数，主要作用有三个：

- 控制成本
- 控制回复长度
- 防止模型输出过长或被无限生成

#### 延伸追问

- 如果设得太小会怎样？

可能导致回答被截断，不完整。

---

### 面试题10：什么是 Token？为什么它在大模型应用里很重要？

#### 标准回答

Token 是大模型处理文本的基本计量单位。模型的上下文长度、输出长度和计费方式，通常都和 token 有关。

它之所以重要，是因为：

- 输入太长会超上下文限制
- 输出太长会增加成本
- 参数控制和性能优化都离不开 token

#### 高分表达

在大模型工程里，token 是比“字符数”或“字数”更真实的资源单位。

---

### 面试题11：模型密钥配置有哪些方式？推荐哪种？

#### 标准回答

常见方式有三种：

- 硬编码
- 环境变量
- `.env` 配置文件

一般不推荐硬编码，因为存在密钥泄露风险。开发阶段通常推荐使用 `.env` 或环境变量；生产环境则更推荐密钥管理系统或平台级配置方案。

#### 面试加分点

可以补一句：

配置的本质目标是做到“代码和敏感信息分离”。

---

## 5. Message 与对话上下文题

### 面试题12：Chat Model 为什么要用消息列表，而不是只传一个字符串？

#### 标准回答

因为 Chat Model 需要处理的不只是单条文本，而是带角色的上下文结构。消息列表可以清晰表达：

- 系统规则
- 用户输入
- 历史回复
- 中间结果

这比单字符串更适合多轮对话和复杂提示设计。

---

### 面试题13：LangChain 中常见的消息类型有哪些？

#### 标准回答

常见消息类型包括：

- `SystemMessage`
- `HumanMessage`
- `AIMessage`

#### 各自作用

- `SystemMessage`：定义系统角色、约束和风格
- `HumanMessage`：代表用户输入
- `AIMessage`：代表模型历史回复

---

### 面试题14：什么是 `SystemMessage`？为什么它重要？

#### 标准回答

`SystemMessage` 用于定义模型在本轮任务中的角色、行为规则、回答风格和边界条件。它相当于顶层指令，常用于告诉模型：

- 你是谁
- 你该怎么答
- 你不能做什么

#### 高分表达

在很多场景里，`SystemMessage` 决定的是模型“如何回答”，而 `HumanMessage` 决定的是模型“回答什么”。

---

### 面试题15：Chat Model 会自动记住之前的对话吗？

#### 标准回答

不会自动长期记住。所谓多轮对话，本质上是开发者把历史消息再次作为输入传给模型，因此模型才看起来“记得之前说过什么”。

如果不传历史消息，模型通常不会记得之前的对话。

#### 追问：那 Memory 模块是干什么的？

Memory 的作用就是帮助管理和组织这些历史上下文。

---

### 面试题16：多轮对话的本质是什么？

#### 标准回答

多轮对话的本质是“显式传递历史上下文”。也就是把前几轮的：

- 用户消息
- AI 消息

重新组织成消息列表，和当前问题一起发给模型。

---

## 6. Runnable 与调用方式题

### 面试题17：什么是 Runnable？为什么它重要？

#### 标准回答

Runnable 是 LangChain 中的一种统一调用协议。很多组件都实现了这个协议，例如：

- Prompt Template
- Chat Model
- Output Parser

它的重要性在于让不同组件拥有统一的调用方式，比如 `invoke`、`stream`、`batch`，从而便于链式组合。

#### 高分补充

正因为这些组件都实现了 Runnable，才可以使用 `prompt | model | parser` 这样的链式表达。

---

### 面试题18：`invoke`、`stream`、`batch` 分别是什么意思？

#### 标准回答

- `invoke`：处理单条输入，完整返回结果
- `stream`：流式返回结果，边生成边输出
- `batch`：批量处理多条输入

#### 适用场景

- `invoke`：最常用、最稳定
- `stream`：适合聊天或实时交互
- `batch`：适合批量任务处理

---

### 面试题19：同步调用和异步调用的区别是什么？

#### 标准回答

同步调用会阻塞当前流程，必须等当前请求完成后才能继续；异步调用可以在等待网络或模型响应时并发处理其他任务，提高整体吞吐效率。

#### 高分表达

异步不是让单个模型更快，而是让系统在 I/O 等待期间更高效地利用时间。

---

### 面试题20：流式输出和非流式输出怎么选？

#### 标准回答

非流式输出一次性返回完整结果，逻辑更简单、结果更完整，适合摘要、抽取、标准接口场景。  
流式输出边生成边返回，更适合聊天机器人、写作助手等强调实时反馈的交互场景。

#### 面试加分点

选择的核心不是“哪个更高级”，而是“业务更看重稳定性还是交互体验”。

---

## 7. Prompt Template 题

### 面试题21：什么是 Prompt Template？

#### 标准回答

Prompt Template 是 LangChain 中用于模板化管理模型输入的组件。它可以把固定提示词和动态变量结合起来，在运行时生成具体 prompt。

它的核心价值是让 prompt 从一次性字符串变成可复用、可维护的工程对象。

---

### 面试题22：为什么不建议直接拼接字符串写 Prompt？

#### 标准回答

直接拼接字符串在简单 demo 里可以工作，但一旦场景复杂，就容易出现：

- 变量混乱
- 重复代码
- 可读性差
- 难维护

Prompt Template 的价值在于把输入参数化、结构化和规范化。

---

### 面试题23：`PromptTemplate` 和 `ChatPromptTemplate` 的区别是什么？

#### 标准回答

- `PromptTemplate` 用于生成普通字符串 prompt
- `ChatPromptTemplate` 用于生成消息列表，适合 Chat Model

如果底层用的是聊天模型，那么大多数场景更推荐 `ChatPromptTemplate`。

#### 一句话记忆

一个生成字符串，一个生成消息结构。

---

### 面试题24：`PromptTemplate` 的常见参数有哪些？

#### 标准回答

常见参数包括：

- `template`：模板正文
- `input_variables`：模板中需要传入的变量
- `partial_variables`：提前固定的一部分变量

#### 追问：`partial_variables` 有什么用？

适合保存那些固定不变但又希望从模板逻辑中抽离出来的内容，比如统一的格式说明或角色设定。

---

### 面试题25：`format()` 和 `invoke()` 的区别是什么？

#### 标准回答

对于 Prompt 模板来说：

- `format()` 返回的是格式化后的字符串
- `invoke()` 返回的是 `PromptValue`，更适合接入 LangChain 的 Runnable 体系

#### 高分理解

`format()` 更偏“字符串渲染”，`invoke()` 更偏“框架运行接口”。

---

### 面试题26：什么是 Partial Prompt？它有什么价值？

#### 标准回答

Partial Prompt 指的是先把模板中的一部分变量固定下来，剩下的变量在运行时再传入。它可以通过 `partial_variables` 或 `.partial()` 实现。

它的价值是：

- 减少重复输入
- 固化系统级规则
- 提升模板复用性

---

## 8. ChatPromptTemplate 与消息模板题

### 面试题27：`ChatPromptTemplate` 适合什么场景？

#### 标准回答

`ChatPromptTemplate` 适合以聊天消息为输入组织方式的场景，尤其适用于：

- Chat Model
- 多角色提示
- 多轮对话
- 需要插入历史消息或中间步骤的复杂场景

---

### 面试题28：常见的消息模板类有哪些？

#### 标准回答

常见的消息模板类包括：

- `SystemMessagePromptTemplate`
- `HumanMessagePromptTemplate`
- `AIMessagePromptTemplate`
- `ChatMessagePromptTemplate`

#### 它们的作用

用于在模板层面生成不同角色的消息对象。

---

### 面试题29：什么是 `MessagesPlaceholder`？它解决了什么问题？

#### 标准回答

`MessagesPlaceholder` 用于在 `ChatPromptTemplate` 中预留一个位置，在运行时插入一串消息列表。

它主要解决：

- 动态插入历史对话
- 动态插入中间执行步骤
- 动态拼接上下文消息

#### 高分表达

它本质上是在 Prompt 模板里预留“消息插槽”。

---

### 面试题30：`MessagesPlaceholder` 最常见的应用场景是什么？

#### 标准回答

最常见的场景包括：

- 多轮对话历史记录
- Agent 的中间步骤
- 工具调用结果回填

它对于复杂聊天系统和 Agent 系统都很关键。

---

## 9. Few-shot 与示例选择题

### 面试题31：什么是 Few-shot Prompting？

#### 标准回答

Few-shot Prompting 指的是在正式问题前先给模型若干示例，让模型学习期望的回答格式、推理方式或映射规则，再处理当前输入。

本质上是“先示范，再让模型照着做”。

---

### 面试题32：Few-shot 有什么作用？

#### 标准回答

Few-shot 的作用主要包括：

- 提升输出格式稳定性
- 引导模型学习特定风格
- 让模型掌握某种输入输出映射关系

尤其适用于：

- 自定义格式生成
- 规则映射
- 风格约束

---

### 面试题33：`FewShotPromptTemplate` 和 `FewShotChatMessagePromptTemplate` 的区别是什么？

#### 标准回答

- `FewShotPromptTemplate` 适用于普通字符串 prompt
- `FewShotChatMessagePromptTemplate` 适用于聊天消息格式的 few-shot 场景

区别本质上在于示例组织形式不同：一个是字符串模板，一个是消息模板。

---

### 面试题34：Few-shot 一定是示例越多越好吗？

#### 标准回答

不是。

示例过多会带来：

- token 成本上升
- 输入变长
- 推理变慢
- 无关示例可能引入干扰

所以示例应该追求质量和代表性，而不是数量越多越好。

---

### 面试题35：什么是 Example Selector？为什么需要它？

#### 标准回答

Example Selector 是用于从大量候选示例中，根据当前输入动态选择最相关示例的机制。

它的意义在于：

- 减少 token 消耗
- 避免把所有示例无差别塞进 prompt
- 提升示例与当前问题的相关性

#### 高分补充

很多 Example Selector 背后会结合 Embedding 和向量相似度检索。

---

## 10. Output Parser 题

### 面试题36：什么是 Output Parser？

#### 标准回答

Output Parser 是 LangChain 中用于解析模型输出的组件，它负责把模型生成的自然语言结果转换成更适合程序处理的结构，比如字符串、JSON、列表、XML 或日期时间对象。

#### 核心价值

让模型输出从“人能看懂”变成“程序能消费”。

---

### 面试题37：为什么说 Output Parser 很重要？

#### 标准回答

因为真实业务系统通常不满足于拿到一段自然语言，它更需要：

- 结构化字段
- 可校验格式
- 可编程对象

如果没有 Output Parser，下游逻辑会非常脆弱，也不利于自动化处理。

---

### 面试题38：`StrOutputParser` 是干什么的？

#### 标准回答

`StrOutputParser` 会把模型输出对象中的文本内容提取出来，转换成普通字符串。

它最适合：

- 简单文本输出场景
- 不需要复杂结构化结果的任务

---

### 面试题39：`JsonOutputParser` 的作用是什么？

#### 标准回答

`JsonOutputParser` 用于把模型输出解析成 JSON 对象或对应的 Python 结构，特别适合结构化输出场景，比如：

- 信息抽取
- API 参数生成
- 数据存储
- 下游流程调用

#### 追问：它能单独工作吗？

不能完全依赖它。通常还需要在 Prompt 中明确要求模型按 JSON 格式输出。

---

### 面试题40：`JsonOutputParser.get_format_instructions()` 有什么用？

#### 标准回答

这个方法会生成一段格式说明，用于指导模型按解析器期待的结构输出结果。

它的价值在于让 Prompt 和 Parser 协同工作，从而提高结构化输出的成功率和稳定性。

---

### 面试题41：为什么“Prompt 约束 + Parser 解析”要一起使用？

#### 标准回答

因为 Parser 只是负责解析，不负责替你完全约束模型行为。如果 Prompt 没有明确输出要求，模型可能返回大量自由文本，导致解析失败。

因此稳定结构化输出一般要靠：

- Prompt 中的格式约束
- 模型的执行能力
- Parser 的解析能力

三者配合。

---

### 面试题42：除了字符串和 JSON，还有哪些常见解析器？

#### 标准回答

常见解析器还包括：

- `XMLOutputParser`
- `CommaSeparatedListOutputParser`
- `DatetimeOutputParser`

此外还有更进阶的：

- `StructuredOutputParser`
- `OutputFixingParser`
- `RetryOutputParser`

---

## 11. 链式组合与工程化题

### 面试题43：如何理解 `prompt | model | parser` 这条链？

#### 标准回答

这条链是 LangChain 中最基础的一种组合方式，分别表示：

- `prompt`：组织输入
- `model`：执行推理
- `parser`：解析输出

它正好对应 Model I/O 的三个阶段，因此是很多复杂应用的最小骨架。

#### 高分表达

这不是简单语法糖，而是 LangChain 组件化和可组合设计的核心体现。

---

### 面试题44：为什么说这条链是后续很多高级能力的基础？

#### 标准回答

因为更复杂的链、RAG、Agent、Memory，本质上仍然是在更复杂地组织输入、调用模型、处理输出。

所以 `prompt | model | parser` 可以看作是 LangChain 应用开发的最小闭环。

---

## 12. 本地模型题

### 面试题45：LangChain 能调用本地模型吗？

#### 标准回答

可以。LangChain 不只支持云端大模型，也支持本地模型，例如通过 Ollama 集成运行本地的开源模型。

---

### 面试题46：什么是 Ollama？

#### 标准回答

Ollama 是一个本地大模型运行与管理平台，能够帮助开发者在本地下载、启动和调用像 Qwen、DeepSeek 等开源模型，并提供比较方便的接口与框架集成。

#### 面试加分点

它的重要意义在于降低本地部署和使用开源模型的门槛。

---

### 面试题47：为什么企业会考虑本地模型而不是只用云模型？

#### 标准回答

常见原因包括：

- 数据隐私和合规要求
- 成本考虑
- 内网或离线场景
- 对底层模型可控性要求更高

#### 一句话总结

本地模型通常不是为了“更炫”，而是为了“更可控”。

---

## 13. 对比与易错题

### 面试题48：`PromptTemplate` 和 `ChatPromptTemplate` 怎么快速区分？

#### 标准回答

快速区分方法是看输出形式：

- `PromptTemplate` 输出字符串
- `ChatPromptTemplate` 输出消息列表

如果底层是聊天模型，后者通常更自然。

---

### 面试题49：`AIMessage` 和普通字符串有什么区别？

#### 标准回答

`AIMessage` 是聊天模型返回的消息对象，里面不只有文本内容，还可能包含：

- 角色信息
- 元数据
- token 使用信息

普通字符串则只是提取后的纯文本。

---

### 面试题50：Chat Model 和 Memory 是一回事吗？

#### 标准回答

不是。

Chat Model 只是支持消息化输入输出，并不自动等于“有记忆”；Memory 是专门用于组织和保存上下文历史的机制。

---

### 面试题51：为什么说“会写 Prompt”不等于“会做 Model I/O”？

#### 标准回答

因为 Model I/O 不只是写提示词，还包括：

- 输入模板设计
- 消息结构组织
- 模型参数设置
- 输出结构化
- 流式或异步调用
- 上下文管理

也就是说，它是完整的交互工程，而不只是提示词技巧。

---

## 14. 综合设计题

### 面试题52：如果让你设计一个最小版 LangChain 问答系统，你会怎么做？

#### 参考回答

我会先构建最小的 Model I/O 链路：

1. 使用 `ChatPromptTemplate` 定义 system 和 human 输入
2. 用 `ChatOpenAI` 或其他聊天模型完成推理
3. 用 `StrOutputParser` 或 `JsonOutputParser` 解析输出

如果需要多轮对话，我会加入 `MessagesPlaceholder` 来插入历史消息；如果需要结构化结果，我会明确设计输出格式并接入 JSON 解析器。

#### 面试加分点

这表明你知道先搭最小闭环，再逐步扩展复杂度。

---

### 面试题53：如果系统需要稳定返回结构化结果，你会怎么设计？

#### 参考回答

我会从三个层面设计：

1. 在 Prompt 中明确要求输出结构
2. 使用 `JsonOutputParser` 或其他合适的解析器
3. 对结果进行校验，必要时加入修复或重试机制

核心目标是让模型输出尽量可预测，并且方便下游程序消费。

---

### 面试题54：如果是多轮对话系统，你会怎么处理上下文？

#### 参考回答

我会将历史消息显式保存，并通过 `MessagesPlaceholder` 或 Memory 机制，在每次调用时把必要的历史消息重新注入到 `ChatPromptTemplate` 中。

同时还会考虑：

- 上下文窗口限制
- 历史裁剪策略
- 摘要压缩策略

---

### 面试题55：如果任务需要实时打字机式输出，你会如何实现？

#### 参考回答

我会使用模型的流式调用方式，例如 `stream`，并结合前端或回调机制逐步消费模型返回的内容块，让用户边等待边看到结果输出。

这类设计更适合聊天机器人和写作助手。

---

## 15. 高频一句话快答

### Q1：Model I/O 的三部分是什么？

Prompt Template、Model、Output Parser。

### Q2：为什么推荐 Chat Model？

因为它更适合消息结构、上下文管理和现代主流模型接口。

### Q3：PromptTemplate 和 ChatPromptTemplate 最大区别是什么？

一个生成字符串，一个生成消息列表。

### Q4：Chat Model 会自动记住上下文吗？

不会，必须把历史消息再传进去。

### Q5：JsonOutputParser 能保证模型一定输出合法 JSON 吗？

不能，通常还需要 Prompt 约束配合。

### Q6：MessagesPlaceholder 是做什么的？

在聊天模板中动态插入消息列表。

### Q7：Few-shot 的核心思想是什么？

先给示例，再让模型照着做。

### Q8：为什么要用 Output Parser？

因为业务系统需要程序可处理的结构化结果。

### Q9：`prompt | model | parser` 代表什么？

输入组织、模型推理、输出解析。

### Q10：Ollama 的意义是什么？

降低本地大模型运行和集成门槛。

## 16. 面试收尾总结模板

如果面试官让你概括这一章，你可以这样说：

Model I/O 是 LangChain 中最基础的交互模块，核心是把大模型调用拆成输入模板化、模型统一调用和输出结构化三个环节。  
在现代大模型开发里，我会优先使用 Chat Model，并结合 `ChatPromptTemplate` 管理消息结构，通过 `MessagesPlaceholder` 组织上下文，通过 `Output Parser` 把结果转成程序可消费的数据。  
从工程角度看，Model I/O 的价值不只是“能调模型”，而是让模型调用变得更规范、更稳定、更适合集成到真实系统里。
