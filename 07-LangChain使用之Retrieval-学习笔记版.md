# 《07-LangChain使用之Retrieval》学习笔记版

## 1. 这章在讲什么

这一章讲的是 `LangChain` 里和“检索增强生成”相关的一整套能力，也就是我们常说的：

- `Retrieval`
- `RAG`
- 私有知识库问答
- 文档检索增强

如果你是 AI 小白，但懂一些计算机基础，那么你可以先把这一章理解成一句话：

`Retrieval = 让大模型在回答问题之前，先去外部资料里查一查，再结合查到的内容作答`

它解决的不是“模型会不会说话”，而是“模型说的话靠不靠谱”。

---

## 2. 为什么需要 Retrieval

### 2.1 大模型的核心问题：幻觉

大模型虽然很强，但它有一个非常典型的问题，叫做 `Hallucination（幻觉）`。

所谓幻觉，不是说模型完全不会，而是：

- 它可能根据统计规律“编”一个看起来合理的答案
- 这个答案语气很自信，但事实不一定对
- 在通用问题上可能还能凑合
- 一到专业领域、私有领域、实时领域，就容易翻车

比如：

- 公司内部制度
- 某个项目的接口文档
- 某个行业的专业知识
- 最新的政策、价格、数据

这些内容往往不在模型原始训练知识里，或者不是它能稳定记住的。

所以，真实业务里不能只靠模型“脑补”。

### 2.2 RAG 的基本思想

`RAG = Retrieval-Augmented Generation`

翻译成人话就是：

`先检索，再生成`

一个典型流程是：

1. 用户提问题
2. 系统先去知识库里找相关内容
3. 把找到的内容作为上下文塞给大模型
4. 大模型基于这些外部证据生成回答

这样做的好处是：

- 让回答更贴近真实资料
- 降低胡编乱造概率
- 让模型能“接入私有知识”
- 不需要频繁微调模型

### 2.3 RAG 的优缺点

优点：

- 比纯 Prompt 更稳定，因为多了真实上下文
- 比微调更灵活，知识更新不需要重新训练模型
- 更适合企业知识库、文档问答、客服系统
- 对私有数据更友好，很多场景只做向量化和检索即可

缺点：

- 响应链路更长，延迟更高
- 要额外消耗检索和嵌入的成本
- 检索结果质量会直接影响最终回答质量
- 如果切块策略差、向量模型差，RAG 也会答偏

一句话总结：

`RAG 不是万能药，但它是当前解决“知识型问答”最实用、最主流的工程方案之一。`

## 3. Retrieval 整体流程

课件把 Retrieval 拆成了一条很完整的流水线：

`Source -> Load -> Transform -> Text Splitting -> Embed -> Store -> Retrieve`

下面逐个拆开说。

### 3.1 Source：数据源

数据源就是外挂给大模型的“外部知识”。

它可以来自：

- txt
- pdf
- csv
- json
- html
- markdown
- 代码文件
- 文件夹
- 网站
- API 返回结果

所以你要先建立一个概念：

`RAG 的本质不是“查数据库”，而是“把各种非结构化知识转成可检索形式”。`

### 3.2 Load：文档加载

文档加载器 `Document Loader` 的作用是把各种来源的数据，统一加载成 `Document` 对象。

这个统一抽象非常重要，因为后面不管是切分、向量化还是检索，都是围绕 `Document` 处理的。

一个 `Document` 至少有两个核心字段：

- `page_content`：真正的正文内容
- `metadata`：元数据，比如来源、页码、作者、标题等

这也是为什么不同 Loader 虽然加载逻辑不同，但最后的使用方式很统一。

你可以把它理解成：

`各种原始文件 -> 统一中间对象 Document -> 再进入后续处理流水线`

### 3.3 Transform：文档转换

文档加载进来后，通常不能直接拿去检索，还要做转换。

常见转换包括：

- 文本拆分
- 去重
- 元数据提取
- 多语言转换
- 对话转问答格式

其中最关键、最常用、最影响效果的，就是：

`Text Splitting（文档切分）`

## 4. 为什么一定要切块

### 4.1 不切块会有什么问题

如果一整篇文档直接拿去做检索和喂给模型，会有两个大问题：

1. 无关信息太多
2. 超出模型上下文窗口

举个例子：

用户只问“LangChain 中 Retriever 是什么”，
但你把一整本 PDF 全塞进 Prompt，那里面大部分内容都和这个问题无关。

这会带来几个后果：

- 噪声太多，模型更难聚焦重点
- Token 成本更高
- 检索命中更粗糙
- 回答容易被大段无关内容干扰

所以 RAG 一般不是以“整篇文档”为最小单位，而是以：

`chunk（文本块）`

作为存储和检索单位。

### 4.2 切块本质上是在做什么

切块其实是在平衡三件事：

- 每块不要太大，否则噪声多、成本高
- 每块不要太小，否则语义被切碎
- 相邻块之间最好保留一点重叠，避免边界信息丢失

所以切块不是一个简单的字符串分割问题，而是：

`一个非常影响召回质量的工程优化问题`

## 5. 文档切分器 Text Splitters

### 5.1 常见切分策略

课件提到的思路主要有：

- 按句子切
- 按固定字符数切
- 按固定字符数 + overlap 切
- 递归字符切分
- 语义切分
- 按 token 切

不同策略没有绝对最好，只有是否适合当前场景。

### 5.2 TextSplitter 的核心参数

理解这几个参数非常重要：

- `chunk_size`
  每个块的最大长度

- `chunk_overlap`
  相邻块之间重叠的内容长度

- `length_function`
  如何计算长度，默认一般是 `len`

- `keep_separator`
  是否保留分隔符，比如句号、换行

- `add_start_index`
  是否记录块在原文中的起始位置

初学者最容易忽略的一点：

`chunk_size 和 chunk_overlap 不是随便填的，它们决定检索粒度、上下文完整性和成本。`

### 5.3 CharacterTextSplitter

这是最基础的字符切分器。

它适合：

- 演示原理
- 简单文本
- 对语义要求不高的粗处理

但它的问题也很明显：

- 太机械
- 容易把一句话切断
- 中文场景下尤其容易切得不好

所以实际项目里，一般不把它当首选。

### 5.4 RecursiveCharacterTextSplitter

这是最常用、最通用的切分器之一。

为什么它常用？

因为它不是简单按固定长度硬切，而是会按一组分隔符递归尝试，比如：

- `\n\n`
- `\n`
- 空格
- 空字符串

它的思路是：

1. 先尽量在自然边界切
2. 如果还是太长，再进一步往更细粒度切
3. 最后实在不行，再退化到字符级切分

这就比纯字符切分更合理。

如果你刚开始学 RAG，默认优先记住一句话：

`不会选切分器时，先从 RecursiveCharacterTextSplitter 开始。`

### 5.5 TokenTextSplitter

大模型的上下文限制和计费，很多时候是按 `Token` 计算，而不是按字符数。

所以按 token 切分的意义在于：

- 更贴近模型真实输入限制
- 更容易控制成本
- 更适合对接 OpenAI / Claude 这类模型

这里你要特别记住：

`字符数 != token 数`

比如中文、英文、数字、标点在不同 tokenizer 里切法都可能不同。

### 5.6 SemanticChunker

这是更高级的“语义切块”思路。

它不是按照字符边界切，而是先借助嵌入模型感知语义，再决定在哪些位置断开。

优点：

- 更保持语义完整
- 更适合高质量检索场景

缺点：

- 计算更贵
- 实现更复杂
- 处理速度更慢

所以它不是“默认必用”，而是：

`当你发现普通切块召回效果不理想时，可以升级尝试语义切块。`

### 5.7 特殊格式切分器

LangChain 还提供了一些针对特殊内容的切分方式：

- `HTMLHeaderTextSplitter`
  按 HTML 标题层级切

- `MarkdownTextSplitter`
  按 Markdown 结构切

- `CodeTextSplitter`
  按代码结构切

这类切分器的核心思想是：

`不要把有结构的信息，当成普通纯文本暴力切开`

比如代码按函数、类切，比按 500 个字符硬切，效果通常好很多。

## 6. 文档加载器 Document Loaders

这章列了很多 Loader，初学者要抓主线，不要被 API 名称淹没。

### 6.1 TextLoader

用于加载 txt 文本。

特点：

- 最简单
- 适合入门
- 可以快速理解 `Document.page_content` 和 `metadata`

### 6.2 PyPDFLoader

用于加载 PDF。

特点：

- 很常见
- 每一页通常会转成一个 `Document`
- `metadata` 里往往能带页码、来源路径等信息

这对后面做“回答引用来源页码”很有帮助。

### 6.3 CSVLoader

适合结构化表格数据。

它通常会把每一行转成一个文档。

可以指定：

- 加载哪些列
- 哪一列作为 `source`

这对做知识库问答、表格描述、商品信息检索很有用。

### 6.4 JSONLoader

JSON 是真实业务里特别常见的数据格式。

这部分有一个很重要的点：

`JSONLoader 往往不是“整个文件直接读进来”这么简单，而是要通过 jq_schema 精准抽取你真正关心的字段。`

比如：

- 只取 `.messages[].content`
- 只取 `.data.items[].content`
- 把 `title + content` 拼起来作为正文

所以初学者要形成一个意识：

`做 JSON RAG 时，最重要的不是会不会 load，而是会不会提炼“什么字段该进 page_content，什么字段该放 metadata”。`

### 6.5 HTML / Markdown / Directory Loader

这些 Loader 的价值在于：

- HTML 适合网页内容
- Markdown 适合技术文档、知识库
- DirectoryLoader 适合批量加载整个目录

这说明 LangChain 的设计目标不是面向某一种文件，而是：

`统一接各种异构数据源`

## 7. BaseLoader 和 Document 的设计意义

课件里还专门讲了源码层面的统一抽象。

### 7.1 BaseLoader

不同 Loader 最终都围绕 `BaseLoader` 这个统一接口设计。

这意味着：

- 外部来源不同
- 内部处理逻辑不同
- 但对上层流程暴露的调用方式尽量一致

这就是框架设计里典型的：

`面向抽象编程`

### 7.2 Document

`Document` 是整个 Retrieval 流水线的核心数据结构之一。

它至少有两个关键属性：

- `page_content`
- `metadata`

这一抽象非常重要，因为后续的：

- 切分
- 向量化
- 存储
- 检索

都围绕它展开。

你可以把 `Document` 理解成：

`LangChain 在“非结构化知识处理”里的通用数据载体`

## 8. 嵌入模型 Text Embedding Models

### 8.1 什么是嵌入

嵌入模型的作用是：

`把文本变成向量`

为什么要变成向量？

因为计算机不理解“语义相近”这种抽象概念，但它能计算：

- 向量距离
- 余弦相似度
- 最近邻搜索

于是文本经过 embedding 后，就可以在向量空间里比较“谁和谁更像”。

### 8.2 embedding 在 RAG 里的位置

RAG 里至少有两处会用到 embedding：

1. 文档入库时，把文本块转为向量
2. 用户提问时，把问题也转为向量

然后通过“问题向量”和“文档向量”的相似度，找出最相关的块。

所以一定要记住：

`检索的本质不是关键字匹配，而是向量空间中的语义相似匹配。`

### 8.3 两个常见接口

LangChain 里通常会区分两类接口：

- `embed_query`
  给查询语句做向量化

- `embed_documents`
  给文档列表做向量化

这两个接口分开设计的原因是：

- 查询和文档有时会走不同优化路径
- 某些模型会针对 query / doc 做不同编码处理

### 8.4 课件里的理解重点

这一章不是深入讲 embedding 算法原理，而是强调它在工程上的角色：

- 让文本可计算
- 让语义匹配成为可能
- 为向量检索提供基础

对于初学者来说，这个层面的理解已经足够支撑你搭建 RAG 系统。

## 9. 向量存储 Vector Stores

### 9.1 什么是向量数据库

当文本被转成向量后，你就需要一个地方存它们，还要支持快速相似度搜索。

这个地方就是：

`Vector Store / Vector Database`

它和传统数据库的区别在于：

- 传统数据库擅长精确匹配
- 向量数据库擅长相似性检索

也就是说，它不是问“完全一样吗”，而是问：

`谁和它最像`

### 9.2 为什么传统数据库不够

如果你想查“和量子力学相关的内容”，传统 SQL 更适合做：

- 精确字段过滤
- 条件匹配

但它不擅长做：

- 语义近似匹配
- 高维向量最近邻搜索

所以向量数据库的价值就在这里。

### 9.3 常见向量库

课件提到的有：

- `Chroma`
- `FAISS`
- `Milvus`
- `Pinecone`
- `Redis`

你可以先这样记：

- `FAISS`：经典、本地、偏检索库
- `Chroma`：入门友好，适合本地实验
- `Milvus / Pinecone`：更偏生产级或服务化

### 9.4 Chroma 的理解

课件里重点演示了 `Chroma`。

它有两种常见模式：

- 内存模式
- 持久化模式

如果指定 `persist_directory`，就会把数据落到磁盘。

这意味着：

- 开发阶段可以先内存跑通
- 需要反复复用索引时再持久化

### 9.5 典型检索方式

课件里展示了几种常用检索：

- `similarity_search`
  直接按相似度检索

- `similarity_search_by_vector`
  直接拿向量去搜

- `similarity_search_with_score`
  返回结果和分数

- `_similarity_search_with_relevance_scores`
  返回相关度得分

- `max_marginal_relevance_search`
  用 MMR 平衡“相关性”和“多样性”

这里你要特别理解：

### 9.6 为什么需要 MMR

如果只按相似度排，前几个结果可能都非常像，内容高度重复。

这样虽然“相关”，但容易冗余。

MMR 的目标是：

- 既要相关
- 又不要全是重复块

这在 RAG 中很重要，因为上下文窗口很贵。

你更想拿到：

- 相关
- 互补
- 覆盖不同角度

的内容，而不是 4 段几乎一样的话。

## 10. 检索器 Retrievers

### 10.1 Retriever 是什么

向量库本身已经能查相似度了，那为什么还要 Retriever？

因为 `Retriever` 是更高层的一层抽象。

它的职责不是存数据，而是：

`按某种召回策略，返回和问题相关的文档`

所以你可以理解成：

- `Vector Store` 偏底层存储与相似搜索能力
- `Retriever` 偏上层召回接口和策略封装

### 10.2 Retriever 的典型步骤

1. 把 query 向量化
2. 去向量库中找最相似的文档块
3. 返回相关块

最常见的获取方式是：

`db.as_retriever()`

### 10.3 常见检索策略

课件里展示了几种：

- 默认相似性检索
- `similarity_score_threshold`
  只有分数超过阈值才召回

- `mmr`
  平衡相关性和多样性

这里的工程含义很强：

如果你的知识库质量一般、噪声很多，阈值检索可以减少误召回；
如果你的内容重复很多，MMR 会更有价值。

## 11. Retrieval 和 LLM 怎么结合

这一章最终要落到“检索增强生成”，也就是：

`Retriever + Prompt + LLM`

基本做法是：

1. 先检索出相关文档
2. 把检索结果拼进 Prompt 的 `context`
3. 再让 LLM 根据这些上下文回答

常见提示词约束是：

- 只允许基于提供内容回答
- 如果上下文没有答案，就明确说不知道
- 不允许编造

这一步的本质是：

`把“检索结果”变成“模型可消费的证据上下文”`

### 11.1 为什么不用 RAG 时也能答

课件里有一个很好的对比：

- 不用 RAG 时，模型也许能答出“北京著名建筑”
- 但那是基于模型参数记忆

如果问题换成：

- 公司内部系统规范
- 某个私有文档里的内容
- 某份 PDF 第几页提到的流程

这时不接外部知识，模型就很可能答不准。

所以：

`RAG 的价值不是替代模型常识，而是补足模型不知道、记不准、容易过时的那部分知识。`

## 12. 这一章的项目主线

课件最后其实在往 Agent 方向延伸，核心思想是：

- 一部分知识来自本地检索器
- 一部分知识来自联网搜索工具
- 再加上聊天历史
- 组合成一个更完整的智能助手

这说明 Retrieval 不只是“知识库问答”本身，它还是 Agent 工具体系里的重要组成部分。

你可以把它理解成：

`Retriever 本质上也是一种工具，只不过它检索的是文档知识。`

## 13. 初学者最容易混淆的几个点

### 13.1 Loader 不等于 Splitter

- `Loader` 负责读进来
- `Splitter` 负责切成块

读进来不代表能直接检索。

### 13.2 Embedding 不等于 LLM

- `LLM` 负责生成答案
- `Embedding Model` 负责把文本变向量

两者常常同时出现，但职责完全不同。

### 13.3 Vector Store 不等于 Retriever

- `Vector Store` 负责存和搜
- `Retriever` 负责按召回策略对外提供检索接口

### 13.4 RAG 不等于微调

微调是把知识“学进模型参数”；
RAG 是把知识“放在模型外部，按需取用”。

两者思路完全不同。

---

## 14. 一条真正应该记住的工程主线

如果把这一章压缩成最核心的工程流程，就是：

1. 准备数据源
2. 用 Loader 读成 `Document`
3. 用 Splitter 切成 chunk
4. 用 Embedding 模型把 chunk 向量化
5. 存进 Vector Store
6. 用 Retriever 根据用户问题召回相关 chunk
7. 把召回结果作为上下文交给 LLM
8. 让 LLM 基于上下文生成回答

这就是最经典的 RAG 主链路。

## 15. 作为 AI 初学者，你应该怎么学这一章

建议按这个顺序掌握：

### 第一步：先把对象关系搞清楚

先搞懂下面这几个对象分别干嘛：

- `Document`
- `Loader`
- `TextSplitter`
- `Embedding`
- `VectorStore`
- `Retriever`
- `LLM`

如果这些角色关系模糊，后面代码会越学越乱。

### 第二步：先会搭最小 RAG

最小可运行链路建议这样学：

1. 用 `TextLoader` 读 txt
2. 用 `RecursiveCharacterTextSplitter` 切块
3. 用 `OpenAIEmbeddings` 做向量
4. 用 `Chroma` 或 `FAISS` 存储
5. 用 `as_retriever()` 检索
6. 把结果拼进 Prompt 给 `ChatOpenAI`

先跑通，再优化。

### 第三步：再研究优化点

重点优化方向通常有：

- 切块策略
- chunk 大小
- overlap 大小
- embedding 模型选择
- 检索策略选择
- Prompt 约束方式
- metadata 设计

真正决定 RAG 效果的，往往不是“有没有接上 LangChain”，而是这些细节。

## 16. 高频总结

### 16.1 一句话总结 Retrieval

`Retrieval 是让大模型在回答前先查资料的工程机制。`

### 16.2 一句话总结 RAG

`RAG = 检索外部知识 + 基于知识生成回答`

### 16.3 一句话总结这一章最重要的能力

`把非结构化文档，变成可检索、可召回、可供大模型利用的知识上下文。`

## 17. 最后给你的学习建议

如果你现在是 AI 小白，请不要一上来就纠结：

- 向量维度是多少
- 余弦相似度公式怎么推
- FAISS 内部索引结构怎么实现

这些当然重要，但不是你现在最先该掌握的。

你现在最该先理解的是：

`为什么要切块、为什么要向量化、为什么要召回、为什么要把召回结果塞给 LLM`

当这条主线真正通了，你就已经从“会调用 API”迈入“理解 RAG 工程链路”了。

这才是这一章最重要的学习目标。
