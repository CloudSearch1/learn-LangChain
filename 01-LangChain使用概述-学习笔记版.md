# 《01-LangChain使用概述》学习笔记版

## 1. 这份课件到底在讲什么

这份 PDF 是 LangChain 入门课的总览篇，核心目标不是教你马上写复杂项目，而是先回答下面几个关键问题：

- LangChain 是什么
- 它和“大模型 API 直调”相比，多解决了什么问题
- 大模型应用开发里最重要的两条路线是什么
- LangChain 的核心组件分别负责什么
- 初学者后面应该怎么学、先学什么、后学什么

如果你是 AI 小白，但有计算机基础，那么可以把这份内容理解为：

`LangChain = 一套围绕大模型应用开发的工程化组装框架`

它本身不是大模型，也不是知识库，更不是“某个具体 AI 产品”。它更像一个中间层或者开发框架，作用是把：

- 提示词
- 大模型调用
- 外部工具
- 检索系统
- 记忆机制
- 工作流

这些东西串起来，让我们更高效地搭建 AI 应用。

一句话类比：

- `LLM 像发动机`
- `LangChain 像整车装配平台`

只有发动机，不等于有一辆能上路的车。

## 2. LangChain 是什么

课件中提到，LangChain 是 2022 年 10 月由 Harrison Chase 发起的开源框架，用来开发由大语言模型驱动的应用。

常见应用包括：

- 对话机器人
- 问答系统
- 文档检索系统
- 企业知识库
- Agent 智能体

### 2.1 不要把 LangChain 和 LLM 混为一谈

这是初学者最容易混淆的一点。

LangChain 不等于大模型，它本身不负责“思考内容”，真正生成文本的是底层模型，比如：

- OpenAI 的 GPT 系列
- Claude
- DeepSeek
- GLM

LangChain 的价值在于“组织流程”。

比如你要做一个企业知识库问答系统，真实流程通常不是只发一句 prompt 给模型，而是：

1. 接收用户问题
2. 去文档库里检索相关资料
3. 把检索结果拼进 prompt
4. 调用模型生成回答
5. 对输出做格式化
6. 记录日志、上下文和调用链路

LangChain 主要就是把这类流程标准化。

### 2.2 为什么说它像 Spring / Django

课件给了一个很好的类比：

- LangChain 之于 LLM，类似 Spring 之于 Java
- LangChain 之于 LLM，类似 Django / Flask 之于 Python

这个类比的本质含义是：

- 你不用框架，也能开发
- 但项目一复杂，就会遇到大量重复工程问题
- 框架的意义不是“替你思考”，而是“帮你规范开发方式”

## 3. 为什么需要 LangChain

### 3.1 直接调大模型 API 不行吗

可以，当然可以。

如果你的任务非常简单，比如：

- 单轮问答
- 一次性文本生成
- 一个很短的 prompt 实验

那么直接用 OpenAI、DeepSeek、Claude 的 API 就够了。

但一旦需求变复杂，问题就来了。

### 3.2 复杂 AI 应用会遇到哪些工程问题

真实 AI 应用通常会涉及：

- Prompt 模板管理
- 多模型切换
- 输出格式约束
- 文档切分与向量化
- 检索增强生成（RAG）
- 工具调用
- 多轮对话上下文
- Agent 决策流程
- 监控、调试、评测

如果全部自己手写，会出现几个典型痛点：

#### 痛点1：不同模型接口不统一

每家模型厂商的 API 细节不同，包括：

- 入参命名不同
- 返回结构不同
- 多模态能力不同
- function calling/tool calling 协议不同

如果业务代码直接绑死某个厂商，后面切模型会很痛苦。

#### 痛点2：复杂流程容易变成“意大利面代码”

比如：

- 先检索
- 再拼 prompt
- 再调用模型
- 再解析结果
- 再调用工具
- 再更新记忆

这些逻辑如果没有统一抽象，很快就会难以维护。

#### 痛点3：大模型应用不是只有“生成”

很多人以为 AI 应用开发就是 prompt engineering，但实际上，工程落地常常比 prompt 更重要。

真正上线的系统还要解决：

- 数据从哪来
- 如何检索
- 如何减少幻觉
- 如何保证输出稳定
- 如何调试失败链路
- 如何评测效果

LangChain 正是在这些地方提供了现成抽象。

### 3.3 使用 LangChain 的核心价值

课件里已经点出了几个非常重要的优势，我这里把它展开成更工程化的理解：

#### 1. 统一抽象

用统一接口来调用不同模型和组件，降低迁移成本。

#### 2. 组件化

把 prompt、model、parser、retriever、memory、tool 拆成可拼装模块。

#### 3. 链式编排

可以把多个步骤连成一条“处理链”，让流程结构更清晰。

#### 4. 更容易搭复杂场景

尤其是：

- RAG
- Agent
- Tool Calling
- 多步工作流

#### 5. 更容易进入生产环境

因为它的生态不只包括运行时，还包括：

- LangSmith：调试、测试、监控
- LangServe：部署成服务
- LangGraph：构建更复杂的 agent 工作流

## 4. LangChain 常见应用场景

课件里给了几个典型项目方向。你可以把这些项目理解为“从简单到复杂”的练级路线。

### 4.1 文档问答助手

技术点：

- Prompt
- Embedding
- RetrievalQA

这是最适合初学者的第一类项目，因为它能让你快速理解 RAG 的核心闭环。

### 4.2 智能日程规划助手

技术点：

- Agent
- Tool
- Memory

它比普通问答更进一步，因为模型不只是“回答”，而是“规划并调用能力”。

### 4.3 LLM + 数据库问答

技术点：

- SQLDatabaseToolkit
- Agent

本质是让模型把自然语言转成数据库操作或 SQL 查询。

### 4.4 多模型路由系统

技术点：

- RouterChain
- 多 LLM

比如某些任务适合便宜模型，某些任务适合强模型，那么就可以做模型路由。

### 4.5 企业知识库助手

技术点：

- 向量数据库
- LLM
- Web/可视化应用框架

这是最常见的落地形态之一，也是很多公司最先做的 AI 项目。

## 5. LangChain 生态结构

这部分是整份课件最重要的“认知地图”之一。

课件将 LangChain 生态分成几个核心组成。

## 5.1 LangChain

这是最基础的应用开发层，主要包括：

- Chains
- Agents
- Retrieval
- Model I/O

此外还包括 `langchain-core` 和 `langchain-community` 等拆分包。

### 你可以这样理解

- `langchain-core`：最底层抽象，像“接口层”和“协议层”
- `langchain`：面向应用开发的主框架
- `langchain-community`：第三方集成集合

如果用后端开发类比：

- core 像规范
- langchain 像主体框架
- community 像插件生态

## 5.2 LangGraph

LangGraph 是在 LangChain 基础上的进一步增强，适合做更复杂的 Agent 编排。

它解决的是这样的问题：

- 一个 Agent 不够怎么办
- 多步骤决策如何建模
- 如何让状态在流程中持续流转
- 如何表达循环、分支、回退、人工介入

所以，LangGraph 更偏：

- 状态机
- 工作流
- 多智能体协作

如果说 LangChain 擅长“搭积木”，那 LangGraph 更像“搭流程图”。

## 5.3 LangSmith

LangSmith 是调试和观测平台。

这在 AI 工程里极其重要，因为大模型应用很难像传统程序那样只靠打断点排查。

LangSmith 主要帮助你：

- 看调用链路
- 看每一步 prompt
- 看模型返回
- 对比不同版本 prompt 效果
- 做测试与评估
- 做线上监控

为什么它重要？

因为 AI 应用失败时，经常不是代码报错，而是：

- 检索结果不准
- Prompt 拼接不对
- 工具调用偏离目标
- 输出格式不稳定

这些问题没有链路追踪工具会非常难定位。

## 5.4 LangServe

LangServe 的作用是把 LangChain 的 Runnable 或链部署成 REST API。

这意味着：

- 你可以把 Python 写的链对外暴露成服务
- Java、前端、其他系统都可以调用
- 更适合真正接入业务系统

这一步体现的是“从 demo 到服务化”的过程。

### 5.5 课件强调的判断

课件认为 LangChain 生态里最有前途的两个方向是：

- LangGraph
- LangSmith

这个判断是有道理的。

因为纯 RAG 已经不是 LangChain 独占优势了，很多框架都能做；而更复杂的 Agent 编排和生产级观测，是 LangChain 更容易拉开差距的地方。

## 7. 环境准备怎么理解

课件中这部分偏操作说明，我这里把它提炼成工程上的关键原则。

### 7.1 Python 版本

推荐 Python 3.10 及以上。

原因不是“越新越好”，而是因为很多 AI 生态包会对 Python 版本有要求，版本过低容易装不上。

### 7.2 虚拟环境很重要

无论你是用：

- `venv`
- `conda`

都建议每个项目独立环境。

原因：

- 避免依赖冲突
- 便于复现
- 便于回滚

### 7.3 pip 和 conda 不要乱混

课件里有一个非常实用的建议：

优先顺序最好是：

1. 先用 conda 安装基础依赖
2. 再用 pip 补充 conda 没有的包

因为混装最容易导致环境不一致。

### 7.4 初学者常见误区

- 误区1：全局环境直接装所有包
- 误区2：装完包不管版本
- 误区3：看到报错就无脑升级全部依赖

正确思路是：

- 固定 Python 版本
- 固定关键包版本
- 每个项目独立环境

## 8. 大模型应用开发的两条主线

这部分是全课重点中的重点。

课件明确指出，大模型应用开发主要有两类核心架构：

- 基于 RAG 的开发
- 基于 Agent 的开发

你可以把它们理解成两条主赛道。

## 8.1 RAG：解决“知识不够新、不够准”的问题

RAG 全称是 `Retrieval-Augmented Generation`，即“检索增强生成”。

### 为什么会需要 RAG

因为大模型本身有两个天然问题：

#### 1. 知识冻结

模型训练完以后，它知道的知识停留在训练截止点，不会自动掌握你公司的最新文档、内部规章、刚发布的产品说明。

#### 2. 幻觉

当模型不知道答案时，它可能不会老老实实说“不知道”，而是生成一个“看起来很像真的”答案。

RAG 的核心思想是：

`先查资料，再回答问题`

### RAG 的标准流程

课件给出的流程非常典型，我把它整理成更容易记忆的版本：

1. 加载原始数据
2. 文本切分
3. 将文本块向量化
4. 存入向量数据库
5. 用户提问
6. 将问题向量化
7. 相似度检索
8. 取回相关片段
9. 把检索结果作为上下文拼进 Prompt
10. 调用 LLM 生成回答

### RAG 里的关键难点

课件特别指出了几个难点，这些都是真实项目中的核心优化点：

- 文件解析
- 文本切块
- 知识检索
- 重排序（Rerank）

其中，最容易被初学者低估的是“切块”。

因为切得太大：

- 检索不准
- token 浪费

切得太小：

- 语义丢失
- 上下文不完整

所以文本切分不是机械操作，而是效果调优核心环节。

### Reranker 是什么

Reranker 可以理解为“二次排序器”。

第一轮检索通常是快速召回，目的是先找出一批“可能相关”的文本；Reranker 再对这些结果做更精细的相关性排序。

适用场景：

- 专业知识库
- 高精度客服
- 法律、医疗、金融等高要求场景

不适合的场景：

- 对延迟非常敏感的实时系统

因为 Reranker 会增加时间成本。

### RAG 的本质

RAG 不是让模型“变聪明”，而是让模型“带着资料答题”。

所以它提升的不是模型底层智力，而是：

- 信息新鲜度
- 回答可依据性
- 领域准确率

## 8.2 Agent：解决“只会回答，不会做事”的问题

Agent 是第二条主线。

课件用一句话概括得很好：

`Agent = LLM + Memory + Tools + Planning + Action`

这说明 Agent 不只是一个聊天机器人，而是一个“能思考、能规划、能调用工具、能执行动作”的系统。

### Agent 的五大核心组成

#### 1. LLM：大脑

负责理解任务、推理、规划。

#### 2. Memory：记忆

负责保存上下文和历史经验。

记忆可以分为：

- 短期记忆：当前会话上下文
- 长期记忆：跨任务保留的知识或经验

#### 3. Tools：工具

模型本身不能直接联网、查数据库、订票、调内部系统，所以需要工具扩展能力边界。

#### 4. Planning：规划

把复杂任务拆解成多个步骤，再逐步完成。

#### 5. Action：行动

不是“想到”就结束，而是把决策真正执行出去。

### Agent 和普通程序的区别

传统程序的特点是：

- 流程由开发者提前写死
- 输入到输出的路径固定

Agent 的特点是：

- 中间过程允许模型自主决策
- 可以根据当前信息动态选择工具和下一步动作

所以 Agent 更灵活，但也更难控。

### Agent 最大的挑战

初学者往往只看到它“很强”，容易忽视它的风险：

- 路径不可预测
- 成本更高
- 延迟更大
- 更容易出现偏航
- 更难做稳定性保障

所以工程上经常不是“能不用 Agent 就不用”，而是：

`只有任务真的需要自主规划时，才考虑 Agent`

## 9. 大模型应用开发的四种方案

课件还总结了四种典型路线，这一部分很适合用来做技术选型。

## 9.1 纯 Prompt

适合：

- 简单生成任务
- 文案改写
- 摘要
- 翻译

特点：

- 最简单
- 最快上线
- 工程成本最低

缺点：

- 无法接入外部知识
- 无法真正执行动作
- 稳定性依赖 prompt 质量

## 9.2 Agent + Function Calling

适合：

- 需要调用天气、地图、数据库、企业系统 API
- 需要任务分解和多步决策

优点：

- 能做复杂任务
- 能接入外部能力

缺点：

- 系统复杂度上升明显

## 9.3 RAG

适合：

- 企业知识库
- 文档问答
- 客服系统
- 政策法规查询

优点：

- 能解决知识更新问题
- 可基于外部证据回答

缺点：

- 需要额外的数据链路和索引系统

## 9.4 Fine-tuning 微调

适合：

- 任务高度稳定
- 需要长期固化风格或能力
- 前面几种方案都无法满足需求

特点：

- 成本最高
- 实施周期长
- 不应该成为默认首选

### 一个重要结论

真实项目中，优先级通常是：

1. 先试纯 Prompt
2. 不够用就上 RAG
3. 再不够用考虑 Tool / Agent
4. 最后才考虑微调

这也是比较符合成本收益比的策略。

## 10. LangChain 的六大核心组件

这是必须掌握的主干知识。

## 10.1 Model I/O

这是使用最频繁、也是最容易上手的模块。

它主要解决三件事：

- 输入怎么组织
- 模型怎么调用
- 输出怎么解析

可以拆成三个子模块理解。

### 1. Prompt / Prompt Template

负责把原始输入组织成适合模型理解的格式。

注意：

Prompt 不只是用户一句话，还包括：

- system 指令
- 用户输入
- 上下文资料
- 输出格式要求

所以 Prompt Template 本质上是“输入装配器”。

### 2. Models

负责以统一接口调用不同模型。

也就是说，LangChain 希望你尽量站在统一抽象层开发，而不是直接深度绑定某个厂商 SDK。

### 3. Output Parser

负责把模型输出解析成指定格式。

例如：

- 字符串
- JSON
- 结构化字段

这对工程开发非常重要，因为业务系统更喜欢结构化输出，而不是一大段自然语言。

### 对初学者的理解建议

Model I/O 是所有后续内容的入口。

如果这块没学明白，后面的：

- Chain
- RAG
- Agent

都会觉得乱。

## 10.2 Chains

Chain 是 LangChain 最重要的概念之一。

它的本质是：

`把多个步骤按顺序连接起来，形成可复用流程`

比如：

`Prompt -> LLM -> Output Parser`

这就是一条最简单的链。

### 为什么 Chain 很重要

因为真实业务通常不是“一次模型调用”就结束，而是多个环节的组合。

如果没有 Chain，你就要手动组织每一步输入输出；
有了 Chain，就可以像搭管道一样拼流程。

### 常见 Chain

- `LLMChain`：最基础调用链
- `SequentialChain`：多个链顺序执行
- `RouterChain`：按任务类型路由到不同链
- `RetrievalQA`：面向检索问答

### 你可以这样理解 Chain

在软件工程里，它有点像：

- 管道模式
- 工作流编排
- 函数式组合

## 10.3 Memory

Memory 用来保存对话历史或上下文。

这是“让模型记住前文”的关键模块。

### 常见 Memory 类型

- `ConversationBufferMemory`：保存完整历史
- `ConversationSummaryMemory`：保存摘要
- `ConversationSummaryBufferMemory`：缓冲 + 摘要结合
- `VectorStoreRetrieverMemory`：把历史放到向量库中检索

### 为什么需要不同类型

因为上下文窗口有限，完整保存全部历史并不总是划算。

所以记忆设计本质上是：

- 信息保留
- token 成本
- 响应效果

三者之间的平衡。

## 10.4 Agents

Agents 是 LangChain 的高阶能力。

它不是简单调用工具，而是：

- 决定什么时候调用工具
- 决定调用哪个工具
- 决定调用完之后下一步做什么

它的关键部分包括：

- `AgentType`
- `Tool`
- `AgentExecutor`

### Agent 的核心价值

让模型从“回答器”升级成“执行者”。

### Agent 的工程代价

它的自由度更高，意味着：

- 可解释性更差
- 测试难度更大
- 稳定性更难保证

所以做 Agent 时一定要重视日志、回放和监控。

## 10.5 Retrieval

Retrieval 对应的就是 RAG 能力。

它包含完整的数据处理链：

- Source：数据源
- Load：加载数据
- Transform：转换与切分
- Embed：向量化
- Store：存储
- Retrieve：检索

### 这一模块的本质

让大模型不再只依赖“训练记忆”，而是能从外部知识里取材料。

### RAG 系统最容易被忽视的一点

很多人把注意力全放在 LLM 上，但在 RAG 项目中，真正决定效果的部分往往是：

- 文档质量
- 切块质量
- 检索策略
- 排序策略

而不是底层模型名字本身。

## 10.6 Callbacks

Callbacks 可以理解为“钩子机制”。

它可以在应用运行的各个阶段插入额外逻辑，比如：

- 记录日志
- 做监控
- 流式输出
- 性能分析

这部分是从“能跑”走向“能运维”的重要一环。

## 11. Hello World 示例在讲什么

课件后半部分给了几个简单示例，虽然代码不复杂，但非常具有代表性。

## 11.1 直接调用大模型

最基础的流程：

1. 读取 `.env`
2. 设置 API Key 和 Base URL
3. 创建 `ChatOpenAI`
4. 调用 `invoke`

这一步的本质是让你先理解：

`LangChain 不是魔法，它底层仍然是在调模型`

## 11.2 使用 Prompt Template

这一步说明：

- Prompt 应该模板化
- 应该有角色区分，如 `system` 和 `user`
- 应该支持变量插入

这能让 prompt 更可维护，而不是把一大段字符串写死在代码里。

## 11.3 使用输出解析器

这一步非常重要，因为实际业务很少只需要“随便一段文本”。

业务经常需要：

- JSON
- 固定字段
- 结构化对象

所以 Output Parser 是“让模型结果变得可编程”的关键能力。

## 11.4 使用向量存储

示例里用了：

- `WebBaseLoader`
- `RecursiveCharacterTextSplitter`
- `OpenAIEmbeddings`
- `FAISS`

这基本覆盖了 RAG 最小闭环：

加载 -> 切块 -> 向量化 -> 存储

## 11.5 使用 RAG

示例展示了一个完整的知识问答过程：

1. 检索相关文档
2. 把检索结果拼进 prompt
3. 强约束模型“只能依据已知信息回答”

这一步体现了 RAG 的关键思想：

`回答不是靠模型自由发挥，而是靠外部证据驱动`

## 11.6 使用 Agent

示例把检索器包装成工具，再交给 Agent 使用。

这个例子很有代表性，因为它说明：

- 检索本身也可以成为工具
- Agent 不一定调用天气 API，也可以调用知识检索工具

这意味着：

`RAG 和 Agent 不是对立关系，而是可以组合`

## 12. 初学者最该抓住的主线

如果你现在刚入门，不要试图一口气把所有模块都学完。

推荐按照下面顺序学。

### 第一阶段：先学 Model I/O

目标：

- 会写 PromptTemplate
- 会调 LLM
- 会用 Output Parser

这是所有后续能力的基础。

### 第二阶段：学 Retrieval / RAG

目标：

- 会加载文档
- 会切块
- 会做 embedding
- 会做向量检索
- 会把检索结果接入 prompt

这是最容易做出“有业务价值项目”的方向。

### 第三阶段：学 Chains

目标：

- 会把多个步骤拼成稳定流程

### 第四阶段：学 Memory

目标：

- 做好多轮对话和上下文管理

### 第五阶段：学 Agents

目标：

- 理解工具调用
- 理解规划与执行
- 理解什么时候该用 Agent，什么时候不该用

### 第六阶段：学 LangSmith / LangGraph

目标：

- 进入生产化和复杂工作流阶段

## 13. 这份课件背后的核心方法论

如果把整份 PDF 再抽象一层，它其实想传达下面这套方法论。

### 13.1 大模型应用开发不是“只会 prompt”

真正的 AI 工程是：

- 模型能力
- 数据能力
- 工程能力
- 监控评测能力

共同构成的。

### 13.2 先解决“可用”，再解决“高级”

不是一上来就做多智能体系统，而是：

1. 先做最小可用链路
2. 再接外部知识
3. 再接工具
4. 再考虑复杂工作流

### 13.3 RAG 和 Agent 是两大主战场

前者解决“知道什么”，后者解决“做什么”。

可以记成：

- `RAG 负责补知识`
- `Agent 负责做决策和执行`

### 13.4 生产级 AI 项目必须重视观测

如果没有链路追踪、测试、评估、监控，那么你的 AI 应用大概率只能停留在 demo 阶段。

## 15. 对初学者的重点提醒

### 15.1 不要把框架学习变成 API 背诵

你真正要理解的是：

- 每个模块解决什么问题
- 模块之间怎么协作
- 什么时候该用，什么时候不该用

### 15.2 先做项目，理解会更快

课件里提到“学习 LangChain 最好的方式就是做项目”，这句话非常对。

建议你第一个项目就做：

- 本地 PDF 知识库问答

这是最容易串起以下知识的项目：

- 文档加载
- 文本切分
- Embedding
- 向量库
- Prompt
- LLM

### 15.3 Agent 不是越早学越好

很多人刚接触 AI 就冲着 Agent 去，但实际上如果：

- Prompt 不会写
- RAG 不会做
- 输出解析不懂

直接学 Agent 会非常痛苦。

### 15.4 版本变化要有心理准备

课件里提到 LangChain 从 v0.1 到 v0.2/v0.3 有明显调整，说明这个生态迭代很快。

所以学习时要区分：

- 稳定概念：RAG、Prompt、Retriever、Memory、Tool
- 易变接口：具体包名、导入路径、类名、调用方式

你应该优先掌握前者。

## 16. 最后总结

这份《LangChain使用概述》本质上是一张“入门地图”。

它最重要的贡献，不是教你写了多少代码，而是帮你建立了三个清晰认识：

### 认识1：LangChain 是工程框架，不是大模型本身

它的作用是把大模型应用中的各种能力标准化、模块化、流程化。

### 认识2：大模型应用开发的两条主线是 RAG 和 Agent

- RAG 解决知识获取问题
- Agent 解决规划和执行问题

### 认识3：真正的 AI 工程不只靠 prompt

还要考虑：

- 数据
- 检索
- 结构化输出
- 工具调用
- 记忆
- 调试与监控

如果你已经掌握计算机基础，那么学完这一章后，最合理的下一步不是继续背概念，而是马上进入：

- Model I/O
- RAG

这两部分的实操学习。
