# 《02-LangChain使用之Model IO》学习笔记版

## 1. 这一章在讲什么

这一章是 LangChain 真正进入“怎么写代码”的第一章。

如果上一章回答的是：

- LangChain 是什么
- 为什么需要 LangChain
- 它有哪些核心模块

那么这一章回答的就是：

- 我们到底怎么和大模型打交道
- 输入应该怎么组织
- 模型应该怎么调用
- 输出应该怎么变成程序可处理的结构

这一章的主题叫 `Model I/O`，名字看起来简单，其实非常核心。

`Model I/O = Model Input + Model Output`

在 LangChain 里，它可以拆成三步：

- `Format`：输入格式化，对应 `Prompt Template`
- `Predict`：调用模型，对应 `Model`
- `Parse`：输出解析，对应 `Output Parser`

你可以把它理解成一条最基础的处理流水线：

`原始需求 -> 提示词模板 -> 大模型 -> 结构化结果`

这条流水线是后面几乎所有高级能力的起点。

无论你以后学习：

- Chains
- Memory
- Retrieval
- Agent

本质上都离不开这三步。

---

## 2. 为什么 Model I/O 是 LangChain 里最重要的入门模块

很多初学者一上来就想学 RAG 或 Agent，但如果 Model I/O 没理解清楚，后面很容易出现两个问题：

### 2.1 只会调用，不会理解

比如你会写：

- `ChatOpenAI(...)`
- `prompt | llm | parser`

但你并不知道：

- prompt 究竟传给模型的是什么
- 返回值为什么有时是字符串，有时是 `AIMessage`
- 为什么同样是问题，有时用 `PromptTemplate`，有时要用 `ChatPromptTemplate`

### 2.2 遇到 bug 不知道卡在哪一层

真实开发里常见的问题有：

- prompt 写得不好，模型答偏了
- 模型参数设置不合理，输出过长或太随机
- 输出不是合法 JSON，解析失败
- 多轮对话没把历史消息带上，导致模型“失忆”

这些都属于 Model I/O 范畴。

所以从工程角度看，Model I/O 是你理解 LangChain 的第一块地基。

---

## 3. Model I/O 的整体认知框架

这一章最核心的思想可以压缩成一句话：

`LangChain 不是直接把你的字符串丢给模型，而是提供了一套“输入规范化 + 调用统一化 + 输出结构化”的机制。`

我们把这个框架拆开看。

## 3.1 输入：Prompt Template

你的原始需求往往不是模型真正接收到的东西。

比如你说：

`帮我解释什么是机器学习`

在真实应用里，模型看到的可能是：

- 角色设定
- 输出格式要求
- 上下文资料
- 用户问题
- 历史对话

所以输入必须被“组织”。

这就是 Prompt Template 的作用。

## 3.2 处理中：Model

LangChain 自己不提供大模型，它做的是统一接入不同模型厂商。

也就是说：

- 模型能力来自 OpenAI、DeepSeek、Claude、Qwen、GLM、Ollama 等
- LangChain 负责用统一抽象把它们接进来

## 3.3 输出：Output Parser

模型天生擅长返回自然语言，但业务程序通常需要的是：

- 字符串
- JSON
- 列表
- XML
- 日期时间

所以输出不能只停留在“看起来像能看懂”，还要“程序能处理”。

这就是 Output Parser 的意义。

---

## 4. LangChain 中模型的分类方式

课件把模型分了几个角度来讲，这个分类法很实用。

## 4.1 按功能分类

主要有三类：

- 非对话模型 `LLMs / Text Models`
- 对话模型 `Chat Models`
- 嵌入模型 `Embedding Models`

### 4.1.1 非对话模型 LLM

它的特点是：

- 输入通常是一个字符串
- 输出通常也是字符串
- 不强调角色消息结构

适合：

- 单轮文本生成
- 摘要
- 翻译
- 单次问答

它的问题是：

- 不原生支持多轮对话上下文
- 不擅长表达 `system / human / ai` 这样的消息角色

所以现代大模型应用开发里，这一类已经不是主流首选。

### 4.1.2 对话模型 Chat Model

这是现在最重要的一类。

它的特点是：

- 输入是消息列表或 `PromptValue`
- 每条消息有角色
- 输出通常是 `AIMessage`
- 更适合多轮对话和复杂上下文

典型角色包括：

- `SystemMessage`
- `HumanMessage`
- `AIMessage`

现代大语言模型应用开发，绝大多数场景都建议以 Chat Model 为主。

### 4.1.3 嵌入模型 Embedding Model

Embedding 模型不负责“回答”，它负责把文本变成向量。

它最常用于：

- RAG
- 相似度检索
- 语义搜索
- 示例选择器

这一章里只是带到，重点会在 Retrieval 章节展开。

---

## 5. 为什么推荐优先使用 Chat Model

这是一个必须理解的结论。

### 5.1 Chat Model 更贴近现代大模型接口标准

现在主流模型厂商大都兼容 OpenAI 风格接口，这种接口核心就是：

- 多消息输入
- 角色化组织
- 对话型返回

所以 Chat Model 更符合当下生态。

### 5.2 Chat Model 更适合工程化应用

真实业务里，输入通常不是一句简单的话，而是：

- 一个 system 指令
- 若干历史对话
- 当前用户问题
- 有时还会夹带工具结果

这些用 Chat Model 表达会非常自然。

### 5.3 即使不是聊天场景，也常用 Chat Model

这一点很多初学者会误解。

“对话模型”不代表只能做聊天。

哪怕你做的是：

- 文章摘要
- 关键词提取
- 信息抽取
- 结构化生成

也依然很常用 Chat Model。

因为它在接口设计上更通用。

## 6. 模型调用时最重要的参数

这一部分很实用，也经常是面试会问的基础题。

## 6.1 必须关心的核心参数

### 1. `api_key`

用于身份认证。

没有它，大多数云模型平台都无法调用。

### 2. `base_url`

指定模型服务地址。

它不是“可有可无”的网址，而是模型接口的服务入口。

### 3. `model` 或 `model_name`

指定你具体调用哪个模型。

这很重要，因为同一个平台上往往有多个模型：

- 便宜模型
- 强能力模型
- 推理模型
- 嵌入模型

### 4. `temperature`

控制输出的随机性。

它不是“模型聪明程度”，而更像“生成多样性旋钮”。

一般可以这样理解：

- 低温度：更稳定、更保守、更接近事实型输出
- 高温度：更发散、更有创意，但更容易跑偏

### 5. `max_tokens`

限制输出最大 token 数。

它的作用包括：

- 控制成本
- 控制响应长度
- 避免输出过长

## 6.2 什么是 Token

Token 是大模型处理文本时的基本计量单位。

你可以暂时把它理解成“模型内部切分后的语言片段”，但它并不完全等于：

- 一个汉字
- 一个单词
- 一个字符

Token 重要的原因有三个：

### 1. 它影响上下文长度

模型一次能处理多少输入，不是看“多少字”，而是看“多少 token”。

### 2. 它影响成本

大多数模型的计费都按 token 来算。

### 3. 它影响输出截断

如果 `max_tokens` 太小，模型可能没说完就被截断。

### 6.3 参数选择的工程建议

课件里给了非常实用的建议，我把它总结成更便于记忆的形式：

#### 事实问答、信息提取

- `temperature` 低一些
- `max_tokens` 够用即可

因为我们希望稳，不希望它自由发挥。

#### 创意写作、头脑风暴

- `temperature` 可以高一些

因为需要更多样性。

#### 客服短回复

- `max_tokens` 可控偏小

#### 长文生成

- `max_tokens` 适当放大

本质上，参数设置是在平衡：

- 成本
- 稳定性
- 生成质量

## 7. 模型配置的三种方式

课件把配置方式分为三类，这其实也是工程规范问题。

## 7.1 方式一：硬编码

就是把 API Key、URL、模型名直接写在代码里。

优点：

- 最快
- 临时测试方便

缺点：

- 极不安全
- 不利于环境切换
- 不利于团队协作

所以只能用于非常临时的 demo。

## 7.2 方式二：环境变量

把敏感配置放到环境变量中，再在代码里读取。

优点：

- 密钥不直接写进代码
- 相对规范

缺点：

- 切终端或换环境时容易失效
- 管理起来不如配置文件直观

## 7.3 方式三：`.env` 配置文件

这是课件推荐、也是初学者最适合的方式。

优点：

- 配置和代码分离
- 使用简单
- 便于开发环境管理

但必须注意：

- `.env` 不能乱提交到公开仓库
- 真实生产环境往往还要接更正规的密钥管理机制

### 一个很重要的安全提醒

课件示例里出现了密钥和代理地址的演示写法。这类内容在学习阶段可以帮助理解，但在真实项目里一定要做到：

- 密钥不进代码仓库
- 不在笔记和截图里暴露
- 使用环境变量或密钥管理方案

## 8. Message 机制是 Chat Model 的核心

这一节是你从“会调模型”进阶到“理解对话模型”的关键。

## 8.1 为什么聊天模型不是收一个字符串就完了

因为聊天模型不是单纯在处理一句文本，而是在处理一段“带角色的对话上下文”。

这意味着输入不再只是：

`"请解释机器学习"`

而可能是：

- system：你是一个专业 AI 老师
- human：请解释机器学习
- ai：前一轮已经给出的回答
- human：那监督学习和无监督学习的区别是什么

所以它天然支持上下文建模。

## 8.2 常见消息类型

### `SystemMessage`

用于定义系统角色、规则、风格、边界。

这是“顶层约束”。

例如：

- 你是一个资深算法工程师
- 你必须用中文回答
- 你只能依据给定信息作答

### `HumanMessage`

代表用户输入。

### `AIMessage`

代表模型历史回复。

这个在多轮对话中非常重要，因为你可以把之前模型说过的话再传给模型，让它“记住”上下文。

## 8.3 多轮对话的本质

初学者容易误以为“聊天模型会自动记住上一次对话”。

其实不是。

模型是否记得上下文，取决于你这次调用时是否把历史消息也带上。

也就是说：

`多轮对话 = 你把历史消息重新放进当前输入`

这是一种“显式上下文传递”，不是模型自己永久记忆。

### 一个关键认知

聊天模型并不会自动拥有长期记忆。

如果你不传历史，它就“忘了”。

这也是后面 Memory 模块存在的原因。

## 9. Runnable 统一调用协议

这一节非常重要，因为它解释了为什么 LangChain 的很多组件可以被链式连接。

课件提到，很多 LangChain 组件都实现了 `Runnable` 协议。

这意味着这些组件共享统一调用接口，比如：

- `invoke`
- `stream`
- `batch`
- `ainvoke`
- `abatch`

## 9.1 为什么这个设计很强

因为它让不同类型的组件拥有一致的使用方式。

比如：

- PromptTemplate 是 Runnable
- ChatModel 是 Runnable
- Output Parser 是 Runnable

所以你才能写出：

`prompt | model | parser`

这其实本质上就是把多个 Runnable 串起来。

## 9.2 常见调用方式

### `invoke`

单次调用，等模型完整返回后再拿结果。

最常用，也最适合入门。

### `stream`

流式输出。

适合聊天、写作助手、需要实时反馈的界面。

### `batch`

批量处理多个输入。

适合批量翻译、批量摘要、批量抽取。

### `ainvoke / abatch`

异步版本。

适合并发调用、提高吞吐量。

---

## 10. 流式输出与非流式输出

这一节看起来像接口细节，实际上和用户体验、系统设计都很有关。

## 10.1 非流式输出

特点：

- 等模型完整生成后一次性返回
- 逻辑简单
- 最稳定

适合：

- 数据抽取
- 摘要
- 标准问答
- 后端接口处理

优点：

- 结构完整
- 更容易统一处理

缺点：

- 用户等待感更强

## 10.2 流式输出

特点：

- 模型边生成边返回 token
- 更像实时对话

适合：

- 聊天机器人
- 写作助手
- 实时交互系统

优点：

- 用户感知更快
- 交互体验更自然

缺点：

- 处理逻辑更复杂
- 对前端和回调机制要求更高

### 设计取舍

如果你的系统更重视：

- 用户体验
- 即时反馈

那流式输出通常更合适。

如果更重视：

- 结构完整
- 后处理方便
- 稳定性

那非流式通常更合适。

## 11. 同步、异步、批量调用怎么理解

这部分很容易被只当成“语法差异”，但它本质上是性能设计问题。

## 11.1 同步调用

特点：

- 一个任务做完再做下一个

优点：

- 简单
- 好理解
- 好调试

缺点：

- 并发效率低

## 11.2 异步调用

特点：

- 遇到等待时不阻塞主流程
- 可以并发处理多个请求

适合：

- 网络请求多
- 并发量大
- 多个模型调用可并行

### 一个本质理解

异步不是让单个模型“更快思考”，而是让等待 I/O 的时间更充分地被利用起来。

## 11.3 批量调用

批量调用适合处理一组相互独立的输入。

例如：

- 一次处理多篇文本摘要
- 一次处理多条分类任务

它的意义在于：

- 提高调用效率
- 减少逐条调度成本

## 12. Prompt Template：为什么必须模板化

这一章的第二大重点就是 Prompt Template。

## 12.1 Prompt 不是“一句话”，而是“输入设计”

初学者常把 Prompt 理解为“问模型的那句话”，这太窄了。

在工程开发里，Prompt 往往包含：

- 角色定义
- 用户问题
- 输出约束
- 上下文信息
- 示例
- 历史对话

所以 Prompt Template 的本质是：

`把输入变成一种可复用、可维护、可组合的程序对象`

## 12.2 PromptTemplate 的作用

`PromptTemplate` 适合生成普通字符串型提示词。

它通常用于：

- 单文本 prompt
- 非聊天模型输入
- 简单模板化场景

你可以把它理解成“给 prompt 做参数化”。

例如把：

`请解释机器学习`

变成：

`请解释{topic}`

这样你就可以复用模板。

## 12.3 为什么不直接拼字符串

因为一旦需求复杂，字符串拼接会迅速失控。

典型问题包括：

- 可读性差
- 变量容易漏
- 不利于复用
- 不利于调试

而 PromptTemplate 至少带来了：

- 明确的输入变量
- 更清晰的模板结构
- 更好的复用能力

---

## 13. PromptTemplate 的关键知识点

## 13.1 `template`

模板正文。

## 13.2 `input_variables`

声明模板里会被替换的变量。

## 13.3 `partial_variables`

用于提前固定一部分变量。

这很有用，因为很多 prompt 中有些内容是“半固定的”，比如：

- 统一风格
- 输出格式说明
- 固定身份设定

你不希望每次都重复传。

## 13.4 `format()` 与 `invoke()`

这是一个非常关键但容易被忽视的点。

### `format()`

返回普通字符串。

适合你只想得到格式化后的 prompt 文本。

### `invoke()`

返回 `PromptValue`。

适合进入 LangChain 的统一 Runnable 流程。

### 怎么理解它们的差异

你可以理解为：

- `format()` 更像“字符串渲染”
- `invoke()` 更像“运行一个可被下游组件继续处理的对象”

---

## 14. 部分提示词模板 Partial Prompt

课件讲了两种方式：

- 初始化时直接通过 `partial_variables`
- 通过 `.partial()` 生成部分模板

这背后的本质思想是：

`把“固定部分”和“变化部分”拆开`

例如：

- 固定：你是一个专业老师，请用中文回答
- 变化：今天要解释什么主题

这样做的好处是：

- 减少重复
- 统一风格
- 更容易维护

在真实项目里，很多“系统级规范”都很适合用 partial 固定。

## 15. ChatPromptTemplate：聊天场景最核心的模板

如果说 `PromptTemplate` 是字符串模板，那么 `ChatPromptTemplate` 就是消息模板。

它适合：

- Chat Model
- 多角色输入
- 多轮对话
- 更复杂的上下文结构

## 15.1 它为什么重要

因为对话模型真正接收的不是一大段纯字符串，而是一个消息列表。

所以你不能只想“prompt 长什么样”，还要想：

- 哪部分是 system
- 哪部分是 human
- 哪部分是 ai

`ChatPromptTemplate` 就是在组织这个消息结构。

## 15.2 常见消息模板

课件提到了这些类型：

- `SystemMessagePromptTemplate`
- `HumanMessagePromptTemplate`
- `AIMessagePromptTemplate`
- `ChatMessagePromptTemplate`

### 如何理解它们

- System：系统规则
- Human：用户输入
- AI：历史回复示例
- ChatMessage：自定义角色消息

### 一个重要工程思想

Prompt 不是单文本，而是消息编排。

你越早建立这个认知，后面学 Agent 和多轮对话就越顺。

## 16. MessagesPlaceholder：为什么它是多轮对话的关键

这部分是实战里非常高频、也非常容易混淆的点。

`MessagesPlaceholder` 的作用是：

`在 ChatPromptTemplate 的某个位置动态插入一串消息列表`

这非常适合：

- 历史对话
- 中间推理步骤
- 工具执行结果

## 16.1 它解决了什么问题

如果没有它，你只能手动把历史消息一条条写死在模板里。

但真实系统里，历史消息是运行时动态变化的。

所以需要一个“占位符”，在执行时把消息列表塞进去。

## 16.2 它最常见的两个用法

### 1. 插入对话历史

比如：

- 之前用户说过什么
- 之前模型答过什么

### 2. 插入中间步骤

比如 Agent 调工具后得到的结果，也可以以消息形式插进去。

### 16.3 一个关键理解

`MessagesPlaceholder` 本质上是在 prompt 模板里预留“上下文插槽”。

这个概念一旦懂了，很多高级链路都会顺很多。

## 17. Few-shot：为什么要给模型示例

这是 Prompt Engineering 中非常经典、也非常实用的一种方法。

## 17.1 什么是 Few-shot

Few-shot 就是在正式问题之前，先给模型几个“输入 -> 输出”的示例，让模型学会你希望的回答方式。

本质上是：

`先示范，再提问`

## 17.2 为什么示例有用

因为很多时候你不只是想让模型“回答对”，还想让它：

- 用固定格式答
- 按特定风格答
- 学会某种映射规则

比如：

- 自定义符号运算
- 特定抽取格式
- 某种风格改写

Few-shot 在这些场景里通常很有效。

## 17.3 两类 Few-shot 模板

### `FewShotPromptTemplate`

适合普通字符串 prompt。

### `FewShotChatMessagePromptTemplate`

适合聊天消息格式。

这两个模板的核心差别，不在“有没有示例”，而在“示例是字符串组织还是消息组织”。

## 17.4 Few-shot 的代价

它不是白来的。

加示例会带来：

- 更多 token 消耗
- 更长输入
- 更高成本
- 有时更慢

所以不是示例越多越好。

关键是：

- 示例要高质量
- 示例要有代表性
- 示例数量要控制

---

## 18. Example Selector：为什么不能把所有示例都塞进去

课件里这一部分其实很高级，但很值得理解。

如果你有很多示例，全部塞进 prompt 会带来两个问题：

- token 成本太高
- 无关示例反而会干扰模型

所以 Example Selector 的思路是：

`根据当前输入，动态挑选最相关的几个示例`

这背后往往会用到：

- Embedding
- 向量相似度
- 向量库

### 本质理解

Few-shot 不一定是“固定例子列表”，也可以是“动态选择最像的例子”。

这是一种更工程化、更节省 token 的做法。

---

## 19. PipelinePromptTemplate 与自定义 Prompt

这部分课件里标注了“了解”，但它代表了一个很重要的方向。

## 19.1 PipelinePrompt 的思想

它本质上是在做：

`多个提示词阶段化拼装`

比如：

1. 先分析问题
2. 再组织检索请求
3. 最后生成答案

这其实是在用 Prompt 构建一个“小工作流”。

虽然课件提到这个类在新版本里已过时，但它传达的思想仍然重要：

`Prompt 也可以分层、分阶段设计，而不一定是一坨大字符串。`

## 19.2 自定义 Prompt 模板

当现有模板类不够用时，可以继承基类自己实现。

这说明 LangChain 的 Prompt 系统不是封闭的，而是可扩展的。

不过对初学者来说，不建议过早钻这一块，先把标准模板用熟更重要。

---

## 20. 从文件加载 Prompt

课件还介绍了从：

- YAML
- JSON

加载 Prompt。

这件事很有工程价值。

### 为什么 prompt 不一定要写死在 Python 代码里

因为在真实项目中，prompt 往往会：

- 频繁修改
- 由产品/运营/算法共同迭代
- 需要版本管理

如果全部写死在代码里，协作成本会很高。

把 prompt 配置化有两个好处：

- 更容易管理
- 更方便和代码逻辑解耦

---

## 21. Output Parser：为什么模型输出不能直接拿来用

这是这一章第三大重点。

## 21.1 模型的天然输出是“自然语言”

自然语言适合人看，但不一定适合程序处理。

例如你想要的是：

- 一个 JSON 对象
- 一个列表
- 一个日期

但模型可能给你：

- 一段解释
- 带 Markdown 包裹的 JSON
- 多了一堆废话

这就会影响后续系统逻辑。

所以输出解析器的本质是：

`把模型返回的自然语言，转换成下游程序更需要的结构`

---

## 22. 常见 Output Parser 的理解

## 22.1 `StrOutputParser`

最简单。

它做的事情通常就是：

`从 AIMessage 里提取 content，变成纯字符串`

适合：

- 简单文本输出
- 不关心复杂结构

### 它的重要意义

它让你意识到：

模型返回对象不等于你真正要的结果。

`AIMessage -> str`

这是第一层解析。

## 22.2 `JsonOutputParser`

这是最重要、最实用的解析器之一。

适合：

- 信息抽取
- 结构化问答
- API 入参生成
- 下游程序处理

### 为什么 JSON 如此重要

因为业务代码更喜欢：

- 键值对
- 结构化字段
- 可校验的格式

而不是一段散文式答案。

### 使用 JSON 解析器时的关键思想

不是“解析器魔法般自动把随便一句话变成 JSON”，而是：

1. 你需要通过 prompt 明确告诉模型输出 JSON
2. 解析器帮助你把结果转换成 Python 结构

### `get_format_instructions()` 的意义

这很重要。

它能自动生成格式要求，并把这些格式要求嵌入 prompt。

本质上是：

`让提示词和解析器协同工作`

这比你手写“请返回 JSON”更规范。

## 22.3 `XMLOutputParser`

适合需要 XML 风格结构化输出的场景。

不过从现代工程实践来看，JSON 往往更常用。

你更应该理解的是它背后的通用思想：

`让模型按指定结构吐结果，再由解析器转成程序数据`

## 22.4 `CommaSeparatedListOutputParser`

适合需要列表输出的场景。

例如：

- 给我 5 个电影分类
- 给我 10 个情绪词

## 22.5 `DatetimeOutputParser`

适合时间格式转换场景。

说明 LangChain 的解析器不仅处理“看起来复杂的大结构”，也能处理具体类型约束。

---

## 23. 为什么“Prompt + Parser”要配合使用

这是一个非常核心的工程思想。

很多初学者以为：

- Prompt 负责提问
- Parser 负责善后

但其实两者应该协同设计。

例如当你要 JSON 输出时，正确思路是：

1. Prompt 中明确格式要求
2. Parser 按这个格式去解析

如果 prompt 不约束，parser 再强也容易失败。

### 一个重要结论

`稳定的结构化输出 = 提示词约束 + 模型能力 + 解析器配合`

不是只靠某一个环节。

---

## 24. LCEL 链式组合是这一章的核心实战形式

课件里多次出现这样的写法：

`prompt | chat_model | parser`

这是 LangChain 表达式语言的一种常见风格。

## 24.1 这条链在干什么

可以拆成：

1. `prompt`：把输入参数格式化为模型输入
2. `chat_model`：执行模型调用
3. `parser`：解析模型结果

这正好对应 Model I/O 三步骤：

- 输入
- 推理
- 输出

### 为什么这种写法重要

因为它把“流程”变成了“可组合的管道”。

这就是 LangChain 的设计精髓之一。

---

## 25. 本地模型：为什么这一章最后讲 Ollama

这部分非常有现实意义。

## 25.1 为什么要学本地模型

云模型有很多优点，但也有现实限制：

- 成本
- 网络访问
- 隐私要求
- 数据合规

很多企业或个人场景会更关心：

- 数据不出本地
- 可以离线运行
- 成本可控

所以本地模型是非常重要的路线。

## 25.2 Ollama 的定位

Ollama 可以理解为：

`本地大模型运行与管理平台`

它帮你做了很多事情：

- 下载模型
- 启动模型
- 提供调用接口
- 方便和 LangChain 集成

## 25.3 在 LangChain 里调用本地模型意味着什么

这意味着 LangChain 不只是云 API 框架，它也可以作为本地模型应用开发框架。

这会带来很强的通用性：

- 同样的 Prompt 思路
- 同样的链式写法
- 只是底层模型来源变了

### 一个重要认知

LangChain 的核心价值不在“它绑定了哪个模型”，而在“它提供了一致的开发抽象”。

---

## 26. 这一章最值得你记住的几个关键区别

## 26.1 `PromptTemplate` 和 `ChatPromptTemplate`

- `PromptTemplate`：生成字符串
- `ChatPromptTemplate`：生成消息列表

如果你用的是 Chat Model，通常更推荐后者。

## 26.2 `format()` 和 `invoke()`

- `format()`：更像字符串渲染
- `invoke()`：进入 Runnable 统一调用体系

## 26.3 `invoke()` 和 `stream()`

- `invoke()`：完整返回
- `stream()`：边生成边返回

## 26.4 `AIMessage` 和 `str`

- `AIMessage`：模型返回的消息对象
- `str`：解析后的普通文本

## 26.5 Chat Model 和 Memory

Chat Model 只是支持消息上下文结构，并不代表它自动拥有记忆。

是否“记住”，取决于你是否把历史消息传进去。

---

## 27. 初学者最容易踩的坑

## 27.1 误以为模型会自动记住之前的聊天

不会。

你每次不传历史，它就像第一次见你。

## 27.2 误以为 JSON 解析器会自动修复所有不规范输出

不会。

你仍然要在 prompt 里给出清晰格式要求。

## 27.3 误以为 temperature 越低越“聪明”

不对。

它主要控制随机性，不等于模型能力强弱。

## 27.4 误以为 Prompt 只是“问一句话”

Prompt 在工程里是输入编排，而不是一句随手提问。

## 27.5 误以为用了 Chat Model 就天然是多轮对话系统

也不对。

多轮对话依赖你管理历史消息。

## 27.6 误以为 Few-shot 一定越多越好

示例太多会增加 token 成本，还可能引入噪声。

## 30. 最后总结

这一章《Model I/O》虽然看起来像“基础语法章”，但其实它决定了你后面学 LangChain 的深度。

如果你真正吃透这一章，你会建立起以下关键认知：

### 1. LangChain 与模型交互不是一句 API 调用那么简单

它包括：

- 输入设计
- 模型调用
- 输出规范化

### 2. Chat Model 是现代大模型应用开发的主流接口

因为它天然适合角色消息、多轮上下文和复杂输入结构。

### 3. Prompt Template 是输入工程化的核心

它让 prompt 从“临时字符串”升级成“可复用模板”。

### 4. Output Parser 是程序化落地的关键

它让模型输出从“人能看懂”走向“程序能消费”。

### 5. `prompt | model | parser` 是最基础也最重要的 LangChain 管道

这条链路几乎会出现在后面所有高级场景里。

如果你现在已经有计算机基础，那么学完这一章后，最应该做的事情不是继续背概念，而是自己亲手练习下面三个最小实验：

1. 用 `ChatPromptTemplate + ChatOpenAI + StrOutputParser` 做一个最小问答链
2. 用 `MessagesPlaceholder` 做一个最小多轮对话
3. 用 `JsonOutputParser` 做一个最小结构化输出

把这三件事做通，你对 LangChain 的理解会进入一个很稳的阶段。
