# 《04-LangChain使用之Memory》学习笔记版

## 1. 这份课件到底在讲什么

这一章讲的是 `LangChain` 里非常核心、也非常容易被初学者误解的一个模块：`Memory`。

很多人第一次接触大模型时，会天然以为：

- 我跟模型聊过一次
- 它下一轮应该“记得我刚才说了什么”

但真实情况不是这样。

大模型默认是“无状态”的。也就是说，模型不会像人一样天然记住你上一轮说过的话。它每次回答，本质上都只是在处理“当前这次你发给它的完整输入”。

所以如果我们想做：

- 多轮聊天
- 连续追问
- 客服对话
- 带上下文的问答
- 能记住姓名、偏好、订单号、病史等信息的 AI

就必须额外设计一个“记忆机制”。

而在 LangChain 中，这个能力就由 `Memory` 模块来承担。

一句话理解：

`Memory = 帮应用保存历史上下文，并在后续调用模型时把有用的历史重新喂给模型`

这句话很重要，因为它说明了一件事：

`Memory 不是让模型自己长出记忆，而是让程序帮模型“补历史”`

---

## 2. 为什么需要 Memory

### 2.1 模型为什么“看起来会聊天”，其实又“不记事”

我们平时使用聊天机器人时，看到的是一个会话框，所以很容易把它想象成一个长期在线、持续思考的智能体。

但从程序执行角度看，大多数大模型接口更像这样：

1. 你传入一段输入
2. 模型基于这段输入生成输出
3. 本轮结束

如果第二轮你什么历史都不传，模型就不知道上一轮发生过什么。

比如：

1. 第一轮问：“我叫小明。”
2. 第二轮问：“我叫什么？”

如果第二轮调用时，没有把“我叫小明”这条历史一起传给模型，那么模型其实并不知道答案。

### 2.2 Memory 解决的本质问题

Memory 的本质不是“存数据”这么简单，而是解决下面这条链路：

`历史对话如何保存 -> 哪些历史需要被取出 -> 用什么形式拼接给模型 -> 本轮结果如何继续写回`

所以它既是存储问题，也是上下文管理问题，更是成本控制问题。

---

## 3. 什么是 Memory

课件里对 Memory 的定义可以概括成：

`Memory 是 LangChain 中用于在多轮对话里保存和管理上下文信息的组件`

这些上下文信息可以是：

- 文本
- 图像
- 音频
- 对话消息
- 实体信息
- 摘要信息
- 检索得到的历史片段

但在这一章里，最主要讨论的还是“聊天历史的记忆”。

### 3.1 Memory 的核心目标

让应用具备以下能力：

- 知道用户刚刚问过什么
- 知道当前对话已经聊到哪里
- 记住用户名字、身份、偏好等关键信息
- 在长对话中保留重要事实，舍弃低价值内容
- 在上下文长度有限的情况下，尽可能维持回答质量

### 3.2 Memory 的底层工作流

课件里给出的设计流程非常重要，建议你把它当作 Memory 的标准执行过程来记：

1. 用户输入当前问题
2. 系统从 Memory 里读取历史消息
3. 系统把“历史消息 + 当前问题”拼成新的 Prompt
4. 把 Prompt 交给 LLM
5. LLM 生成结果
6. 系统把“本轮用户输入 + 模型输出”写回 Memory

所以一个接入了 Memory 的链，通常会和 Memory 交互两次：

- 第一次：`读`
- 第二次：`写`

这是面试和理解源码时都特别关键的一点。

---

## 4. 不用 LangChain，能不能自己实现记忆

能。

课件专门演示了一种最朴素的方法：手动维护 `messages` 列表。

思路很简单：

1. 初始化一个消息列表
2. 用户每提一个问题，就把用户消息 append 进去
3. 模型每回复一次，就把 AI 消息 append 进去
4. 下一轮调用时，把整个消息列表一起传给模型

这就是最基础的“上下文记忆”。

### 4.1 为什么这种方式能工作

因为对聊天模型来说，你传入的并不是一条孤立字符串，而是一串消息：

- `SystemMessage`
- `HumanMessage`
- `AIMessage`

只要历史消息还在列表里，模型就能“看到”前文。

### 4.2 这种方式的问题

虽然简单，但很快会暴露出工程问题：

- 对话越长，传给模型的上下文越大
- Token 消耗越来越高
- 响应速度会变慢
- 可能超过模型最大上下文长度
- 很多旧对话其实已经没价值，但你还在继续传

所以真正的重点不是“能不能记”，而是“怎么高质量地记”。

这也就是 LangChain 各类 Memory 设计出来的原因。

---

## 5. Memory 模块的设计思路

课件中给出了一个非常清晰的演进路径：

### 5.1 第一层：保留全部聊天记录

最直接的方法就是全部保存。

优点：

- 实现最简单
- 信息最完整

缺点：

- 上下文会越来越长
- 成本越来越高

### 5.2 第二层：只保留最近 k 轮

只保留最近几轮交互。

优点：

- 成本可控
- 适合短期上下文依赖

缺点：

- 可能丢掉早期重要信息

### 5.3 第三层：把过去内容压缩成摘要

不再保留所有原文，而是把旧对话总结成一段更短的摘要。

优点：

- 节省 Token
- 保留核心语义

缺点：

- 摘要可能丢细节
- 摘要质量依赖模型

### 5.4 第四层：按实体提取和组织信息

比如把对话里的“人名、地点、药物、订单号、偏好”这些关键信息单独提出来。

优点：

- 对关键事实更稳定
- 更适合高风险场景

缺点：

- 设计更复杂
- 更依赖结构化方案

你可以把它理解成一条进化路线：

`全量记录 -> 滑动窗口 -> 摘要压缩 -> 结构化记忆`

## 6. ChatMessageHistory：最基础的消息存储器

### 6.1 它是什么

`ChatMessageHistory` 是一个非常底层的类，用来存储和管理对话消息对象。

它的定位不是“智能记忆策略”，而是“消息仓库”。

也就是说，它主要负责：

- 存消息
- 取消息
- 按消息对象形式保存历史

它不负责：

- 自动裁剪
- 自动摘要
- 自动挑选最重要的信息

### 6.2 它存的是什么

它存的是消息对象，比如：

- `HumanMessage`
- `AIMessage`
- `SystemMessage`

所以它比简单字符串更结构化。

### 6.3 适合理解成什么

如果你有计算机基础，可以这样类比：

- `ChatMessageHistory` 像一个只负责存消息的容器
- 上层各种 Memory 类，像基于这个容器实现出来的不同缓存策略

### 6.4 它的价值

它特别适合两个场景：

- 你想自己控制消息管理逻辑
- 你想把历史消息直接传给聊天模型

### 6.5 初学者要记住的点

- 它是“基础存储类”
- 它保存的是“消息对象”
- 它不等于完整的 Memory 策略

---

## 7. ConversationBufferMemory：完整保留全部对话

### 7.1 它是什么

`ConversationBufferMemory` 是最经典、最容易理解的一种 Memory。

它会按照原始顺序保存完整对话历史，不做裁剪、不做压缩。

一句话理解：

`ConversationBufferMemory = 原样累计所有历史`

### 7.2 它适合什么场景

适合：

- 对话轮次不多
- 必须保留完整上下文
- 教学示例
- 小型聊天机器人

不太适合：

- 很长的对话
- 高并发系统
- 成本敏感场景

### 7.3 它的两个关键参数

#### `return_messages`

决定 Memory 返回的格式：

- `False`：返回拼接后的字符串
- `True`：返回消息对象列表

如果你使用的是 `ChatPromptTemplate + MessagesPlaceholder` 这类消息式 Prompt，通常更适合 `return_messages=True`。

#### `memory_key`

默认是 `history`，表示历史内容在 Prompt 里使用的变量名。

如果你的 Prompt 写的是 `{chat_history}`，那就需要把 `memory_key` 改成 `chat_history`。

### 7.4 这里最容易踩的坑

Prompt 模板变量名必须和 Memory 输出变量名对齐。

比如：

- Prompt 里写 `{history}`
- Memory 默认 `memory_key="history"`

这就能正常工作。

但如果：

- Prompt 里写 `{chat_history}`
- Memory 还是默认 `history`

那链就对不上。

### 7.5 它的本质优缺点

优点：

- 简单
- 透明
- 最容易调试
- 不会丢上下文

缺点：

- 历史越多，Prompt 越长
- Token 消耗会持续上涨
- 长对话容易超出上下文窗口

所以你可以把它视为：

`最基础、最直观，但不适合无限增长场景的方案`

---

## 8. ConversationChain：对“对话链 + BufferMemory”的封装

### 8.1 它是什么

`ConversationChain` 本质上可以看成是：

`LLMChain + ConversationBufferMemory + 默认对话提示模板`

它的目标是简化初始化，让你更快搭建一个能进行多轮对话的链。

### 8.2 它解决了什么问题

如果你自己写：

- Prompt
- LLM
- Memory
- Chain

会比较啰嗦。

`ConversationChain` 把这些常用部分封装起来，尤其适合入门。

### 8.3 为什么它对初学者重要

因为它让你先理解“记忆型对话链”是什么，而不必一开始就陷入太多底层拼装细节。

### 8.4 要注意的细节

课件里特别提到：

- `ConversationChain` 默认使用的输入 key 通常是 `input`
- 如果你传别的 key，可能会报错

这说明在 LangChain 中，很多组件虽然都和“输入”有关，但它们期待的字段名并不总是一样。

初学者一定要养成一个习惯：

`写链之前先确认：Prompt 变量名、Memory key、invoke 参数名是否一致`

---

## 9. ConversationBufferWindowMemory：只记最近 K 轮

### 9.1 为什么要有它

`ConversationBufferMemory` 会无限累积历史，这在长对话里代价非常高。

但很多时候，模型真正需要的并不是“全部历史”，而只是最近几轮。

于是就有了：

`ConversationBufferWindowMemory`

### 9.2 它的核心思想

只保留最近 `k` 个交互窗口。

这里的重点是：

不是保留最近 `k` 条单独消息，而是保留最近若干轮交互。

这样能让上下文大小更稳定。

### 9.3 它适合什么场景

适合：

- 聊天过程关注短期上下文
- 用户最近几轮问题关联最强
- 希望严格控制历史长度

不适合：

- 早期信息很重要，但后续仍可能被问到

### 9.4 典型问题：为什么 k 太小会“失忆”

课件中就展示了这个现象。

比如先说：

- “我叫孙小空”
- 后面又聊了很多别的
- 如果窗口只保留最近 1 轮
- 最后问“我叫什么？”

那么名字可能已经被窗口裁掉，模型就答不出来。

这说明：

`窗口记忆的本质是用“近期性”换“成本控制”`

### 9.5 怎么理解它的优缺点

优点：

- 成本稳定
- 实现简单
- 很适合短期连续追问

缺点：

- 对长期事实记忆不稳定
- 容易遗忘早期关键信息

---

## 10. ConversationTokenBufferMemory：按 Token 数量裁剪

### 10.1 它和 WindowMemory 有什么不同

`ConversationBufferWindowMemory` 是按“轮数”裁剪。

`ConversationTokenBufferMemory` 是按 `Token` 数量裁剪。

这是一种更贴近模型真实成本的控制方式。

### 10.2 为什么按 Token 裁剪更合理

因为不同消息长度差异很大：

- 一条“你好”很短
- 一条长段落很长

如果只按轮数控制，可能：

- 看起来只保留了 2 轮
- 但这 2 轮文本极长
- 依然可能非常耗 Token

按 Token 控制则更精确。

### 10.3 它的核心参数

`max_token_limit`

表示历史上下文最多保留多少 Token。

一旦超限，旧内容就会被裁掉。

### 10.4 它的价值

这是工程上很实用的一类方案，因为它比“固定保留 3 轮”更符合模型调用成本。

### 10.5 但它也有局限

它只是更精确地“截断”，不是“更聪明地保留信息”。

也就是说：

- 它控制成本很强
- 但并不理解什么信息更重要

如果重要信息恰好出现在很早以前，也一样可能被删掉。

---

## 11. ConversationSummaryMemory：用摘要代替原文

### 11.1 它为什么重要

前面两类方案本质上都属于“裁剪”：

- 按轮数裁
- 按 Token 裁

但裁剪有一个天然问题：

`旧内容一旦被删，就真的没了`

而 `ConversationSummaryMemory` 提供了第三种思路：

`旧内容不直接删掉，而是压缩成摘要`

### 11.2 它是什么

它会利用 LLM 自动对历史对话做总结，把原始长对话变成一段较短的摘要文本。

于是，后续模型看到的不是全部历史原文，而是“历史摘要”。

### 11.3 它适合什么场景

适合：

- 对话很长
- 需要保留整体脉络
- 不要求每个细节都逐字保留

例如：

- 长时间客服会话
- 导师式对话
- 学习辅导
- 长周期项目助手

### 11.4 它的优点

- 能保留长期上下文主线
- 比原始完整历史更省 Token
- 比硬截断更不容易完全忘记早期主题

### 11.5 它的风险

摘要不是原文，存在信息损失。

它可能出现：

- 细节被省略
- 事实被模糊化
- 重点偏移

所以你一定要理解：

`摘要记忆提升的是“压缩后的持续性”，不是“百分百准确复刻”`

### 11.6 为什么这对初学者很关键

很多人一看到“摘要记忆”，就以为这是完美方案。

其实不是。

它只是从“删掉历史”升级到了“压缩历史”，本质上仍然是在做取舍。

---

## 12. ConversationSummaryBufferMemory：摘要 + 最近原文 的折中方案

### 12.1 它为什么常被认为更实用

如果只用 `ConversationSummaryMemory`，最新几轮对话也可能被过度抽象。

但真实对话里，最近消息通常最重要，而且往往需要保留原始细节。

所以 LangChain 提供了混合方案：

`ConversationSummaryBufferMemory`

### 12.2 它的核心思想

把历史分成两部分：

- 较早的内容：做摘要
- 最近的内容：保留原文

你可以把它理解成：

`旧内容压缩存档 + 新内容原样保留`

### 12.3 这类方案为什么很实用

因为它同时兼顾了两件事：

- 长期上下文不要彻底丢
- 最近细节不要被摘要抹平

所以它比“纯 Buffer”和“纯 Summary”都更平衡。

### 12.4 典型应用场景

课件里给了一个客服示例，这个场景非常适合它：

- 用户先问订单状态
- 后来补充下单时间
- 又问能否加急
- 再修正订单号
- 最后追问退货政策

这种对话特点是：

- 历史会越来越长
- 部分早期信息要保留
- 最近消息又很关键

这正是混合记忆的强项。

### 12.5 你应该怎么记它

如果面试或实战要你选一个“默认较稳妥”的 Memory 类型，很多场景下它会比单纯 Buffer 更值得优先考虑。

---

## 13. ConversationEntityMemory：按“实体和事实”来记

### 13.1 它解决了什么问题

摘要虽然比直接裁剪强，但它还是自然语言文本。

自然语言有一个问题：

`模型需要自己再去读懂摘要`

如果摘要里同时包含很多内容，模型可能关注到了“头痛”“换药”，却忽略了“青霉素过敏”这种高风险事实。

于是就有了：

`ConversationEntityMemory`

### 13.2 它是什么

它会尝试从对话中识别出实体，并记录实体的属性和关系。

比如：

- 人名
- 地点
- 药物
- 订单号
- 产品名
- 组织名

### 13.3 为什么它更适合高风险事实

课件用了医疗场景举例，这个例子特别有代表性。

如果用户说：

- 我头痛
- 血压 140/90
- 正在吃阿司匹林
- 对青霉素过敏

如果只是摘要记忆，模型可能在后续建议时漏掉“青霉素过敏”。

但如果把这些信息提成结构化实体：

- 症状：头痛
- 血压：140/90
- 当前用药：阿司匹林
- 过敏药物：青霉素

那程序就可以显式检查。

这说明：

`EntityMemory 的价值不只是“让模型记住”，更是“让程序有机会强约束关键事实”`

### 13.4 它和 SummaryMemory 的本质区别

`ConversationSummaryMemory`

- 结果是自然语言摘要
- 依赖模型再次阅读和理解

`ConversationEntityMemory`

- 结果更接近结构化事实
- 程序更容易针对关键字段做校验

### 13.5 它适合什么场景

适合：

- 医疗
- 客服
- 金融
- 法律
- 用户画像
- 任何“某些字段绝不能漏”的场景

---

## 14. ConversationKGMemory：把记忆组织成知识图谱

### 14.1 它比 EntityMemory 多做了什么

`ConversationEntityMemory` 关注的是“有哪些实体”。

`ConversationKGMemory` 更进一步，关注：

`实体之间是什么关系`

它会把信息组织成三元组：

- 头实体
- 关系
- 尾实体

例如：

- 山姆 -> 是 -> 我的朋友
- 山姆 -> 最喜欢的颜色是 -> 红色

### 14.2 它的价值

一旦记忆被组织成图结构，就更适合：

- 关系推理
- 事实关联
- 复杂人物关系记忆

### 14.3 适合怎么理解

你可以把它看成：

`从“保存对话”升级到“保存知识关系”`

### 14.4 适合什么场景

适合：

- 人物关系复杂的对话
- 长期设定型角色系统
- 需要关系链推理的应用

但它通常比基础 Buffer/Window 更复杂，也更偏进阶。

---

## 15. VectorStoreRetrieverMemory：把历史当作可检索语义记忆

### 15.1 它为什么很特别

前面的几类 Memory，大多还是按照“时间顺序”来管理对话。

而 `VectorStoreRetrieverMemory` 走的是另一条路线：

`不是按时间线拿历史，而是按语义相似度找相关历史`

### 15.2 它怎么工作

大体流程是：

1. 把历史对话转成文本片段
2. 通过 Embedding 模型转成向量
3. 存入向量数据库
4. 当新问题到来时
5. 用新问题去检索语义上最相关的历史片段
6. 再把检索结果提供给模型

### 15.3 这和普通 Buffer 的本质区别

普通 Buffer 依赖“最近说过什么”。

向量记忆依赖“语义上相关什么”。

这意味着：

- 即使某条信息说得很早
- 只要和当前问题语义相关
- 仍然可能被检索回来

### 15.4 它适合什么场景

特别适合：

- 长期记忆
- 用户偏好记忆
- 跨时间跨度的事实召回
- 更像“记忆库”而不是“聊天缓存”的系统

### 15.5 它的局限

它的效果依赖于：

- embedding 质量
- 切分方式
- 检索参数
- 向量库质量

所以它比简单 Buffer 更强，但也更工程化。

## 17. 初学者最应该建立的几个认知

### 17.1 Memory 不是数据库的同义词

Memory 虽然涉及存储，但重点不是“把内容存下来”这么简单，而是：

- 存什么
- 什么时候读
- 什么时候写
- 用什么形式给模型
- 怎样在成本和效果之间做平衡

### 17.2 Memory 本质上是在做“上下文工程”

你可以把 Prompt Engineering 看成“本轮怎么说”。

那 Memory 更像是“历史信息怎么管理并注入”。

### 17.3 记得越多，不一定越好

这是很多新手最容易犯的误区。

因为：

- 更多历史 = 更多 Token
- 更多历史 = 更多噪声
- 更多历史 = 更高成本

真正高质量的记忆，不是无限堆历史，而是：

`把对当前任务真正有帮助的信息留下来`

### 17.4 记忆方案一定和业务绑定

不要问“哪种 Memory 最好”，应该问：

`对我的业务来说，哪种记忆方式最合适`

比如：

- 简单闲聊：Buffer 就够了
- 长客服会话：SummaryBuffer 更合适
- 医疗问诊：EntityMemory 更关键
- 长期偏好记忆：VectorStoreRetrieverMemory 更值得考虑

## 20. 本章总结

你可以把这一章浓缩成下面几句话：

### 20.1 一句话总结

`LangChain Memory 的本质，是帮应用管理历史上下文，并在后续调用中把合适的历史重新提供给模型。`

### 20.2 核心主线

- 模型默认不会自己记住历史
- 多轮对话的“记忆”本质上是程序在管理上下文
- Memory 模块会在链执行时先读历史，再写回新结果
- 不同 Memory 的差别，核心在于“怎么保留历史、怎么压缩历史、怎么召回历史”

### 20.3 各类 Memory 的一句话定位

- `ChatMessageHistory`：底层消息存储器
- `ConversationBufferMemory`：保存全部历史
- `ConversationChain`：对话链快捷封装
- `ConversationBufferWindowMemory`：只保留最近 K 轮
- `ConversationTokenBufferMemory`：按 Token 上限裁剪
- `ConversationSummaryMemory`：把历史压缩成摘要
- `ConversationSummaryBufferMemory`：旧内容摘要，最近内容保留原文
- `ConversationEntityMemory`：提取关键实体和事实
- `ConversationKGMemory`：把实体关系组织成知识图谱
- `VectorStoreRetrieverMemory`：按语义相关性检索历史

### 20.4 站在工程角度最重要的结论

没有一种 Memory 能在所有场景下都最优。

真正成熟的做法，是根据业务场景在下面几件事之间做取舍：

- 完整性
- 成本
- 上下文长度
- 事实稳定性
- 召回准确率
- 工程复杂度

如果你能真正理解这几个取舍，这一章就算学透了。
