# 《03-LangChain使用之Chains》学习笔记版

## 1. 这一章在讲什么

这一章的主题是 `Chains`，也就是“链”。

如果上一章 `Model I/O` 讲的是：

- 如何构造输入
- 如何调用模型
- 如何解析输出

那么这一章回答的是一个更进一步的问题：

`当一个任务不再是“一次模型调用”就能完成时，应该怎么把多个步骤组织成一个稳定流程？`

这就是 Chain 存在的意义。

你可以把这一章理解为：

`LangChain 从“单次调用模型”升级到“搭工作流”的第一步`

所以这一章非常关键，因为它是你从“会调模型”走向“会组织 AI 业务流程”的起点。

---

## 2. 什么是 Chain

课件给出的定义很直接：

Chain 就是把多个组件连接起来，形成一个可复用工作流。

这些组件可以包括：

- Prompt Template
- LLM / ChatModel
- Output Parser
- Memory
- Tool
- 外部数据

### 2.1 用人话怎么理解

单个组件像一个零件，Chain 像一条装配线。

例如你要做一个“解释概念并总结”的任务，流程可能是：

1. 第一步：让模型详细解释
2. 第二步：让模型基于上一步内容做摘要

这已经不是一次调用，而是两个步骤串联。

这时候就需要 Chain。

### 2.2 Chain 的本质

Chain 的本质不是“某个类”，而是一种思想：

`把多个原本独立的处理步骤，组织成一个连续、可复用、可执行的数据流。`

从软件工程角度看，它有点像：

- 管道模式
- 工作流编排
- 数据流处理

---

## 3. 为什么需要 Chain

很多初学者会问：

“我直接写几行代码，一个 prompt 调完再调下一个 prompt，不也能做吗？”

当然可以。

但项目一复杂，你会遇到几个问题。

## 3.1 手工串联步骤很快会变乱

比如你手写：

- 先 format prompt
- 再调用 model
- 再 parse
- 再把结果传给第二个 model
- 再做下一步

当链路变长时，代码会越来越像“流程拼接脚本”，维护成本很高。

## 3.2 不同组件调用方式不统一

课件专门讲了这一点。

如果没有统一协议，你可能会遇到这种情况：

- prompt 用 `.format()`
- model 用 `.generate()`
- parser 用 `.parse()`
- tool 用 `.run()`

这会导致组合时要频繁做适配。

## 3.3 真实业务通常不止一步

典型场景包括：

- 先翻译，再总结
- 先生成 SQL，再执行查询
- 先检索文档，再做问答
- 先写大纲，再写评论

这些都属于多步骤流程。

所以 Chain 的存在，解决的是：

`从“单组件调用”到“多步骤 AI 工作流”的工程化组织问题。`

---

## 4. LCEL：现代 LangChain 里最重要的链式表达方式

这章最核心的知识点之一就是 LCEL。

LCEL 全称：

`LangChain Expression Language`

它是一种声明式的链式表达方式，用 Python 的 `|` 运算符把不同组件组合起来。

## 4.1 LCEL 最基本的形式

课件里给出的标准最小链是：

`prompt | model | output_parser`

这对应了最基础的三步：

- 输入组织
- 模型调用
- 输出解析

### 一个非常重要的理解

LCEL 不是“花哨写法”，而是 LangChain 新式链设计的核心思路。

它让工作流表达得更像“数据流”，而不是一堆零散函数调用。

## 4.2 为什么 `|` 这个符号这么关键

你可以把它理解成 Unix Shell 里的管道符。

含义是：

`前一个组件的输出，作为后一个组件的输入`

也就是说：

1. prompt 接收输入字典
2. prompt 输出 PromptValue
3. model 接收 PromptValue
4. model 输出 AIMessage 或文本
5. parser 接收模型输出
6. parser 输出你真正需要的结果

这就形成了一条可执行的数据处理管道。

---

## 5. Runnable：为什么 LCEL 能把不同组件串起来

这一章另一个必须吃透的点是 `Runnable`。

## 5.1 Runnable 是什么

Runnable 是 LangChain 定义的一套统一调用协议。

只要一个对象实现了这套协议，它就能以统一方式参与 LCEL。

常见实现 Runnable 的组件包括：

- Prompt Template
- Chat Model
- Output Parser
- Retriever
- Agent

## 5.2 它解决了什么问题

它解决的是“调用方式不一致”的问题。

课件举了一个非常典型的对比：

### 没有统一协议时

- prompt 可能用 `.format()`
- model 可能用 `.generate()`
- parser 可能用 `.parse()`

### 有了 Runnable 之后

统一成：

- `.invoke()`
- `.batch()`
- `.stream()`

这样一来，组件就更容易组合了。

## 5.3 Runnable 的工程意义

这不是一个小语法糖，而是框架设计层面的统一抽象。

它带来三个好处：

### 1. 一致性

不同组件统一调用方式。

### 2. 可组合性

可以自然地通过 LCEL 串起来。

### 3. 可扩展性

后续新组件只要遵守 Runnable 协议，就能融入现有链体系。

---

## 6. 最小 Chain：从分步调用到 LCEL 管道

这一节虽然简单，但特别重要，因为它让你看清“链”和“普通代码”到底差在哪。

## 6.1 分步调用

分步调用的思路是：

1. 手动调用 prompt
2. 手动调用 model
3. 手动调用 parser

这当然没错，而且很适合调试。

## 6.2 LCEL 管道式调用

把上面三步压缩为：

`chain = prompt | model | parser`

再统一：

`chain.invoke({...})`

### 这带来的本质变化

你不再手动搬运中间结果，而是把流程整体抽象成一个“可执行链”。

这一步对理解 LangChain 很关键。

### 一个初学者必须有的认知

Chain 不是把几行代码“缩写”一下，而是把流程“对象化、可复用化”。

---

## 7. Legacy Chains 和 LCEL 新式链

课件把 Chains 分成两大部分：

- 传统 Chain
- 基于 LCEL 的新式链

这个区分非常重要。

## 7.1 Legacy Chains 是什么

也就是 LangChain 早期常见的链类，比如：

- `LLMChain`
- `SimpleSequentialChain`
- `SequentialChain`
- `LLMMathChain`
- `RouterChain`
- `StuffDocumentsChain`

这些类在很多老教程里很常见。

## 7.2 新式链是什么

新式链更强调：

- Runnable 协议
- LCEL 表达式
- 工厂函数创建链

例如：

- `create_sql_query_chain`
- `create_stuff_documents_chain`

### 一个重要结论

现在学习 LangChain，不能只学 Legacy Chains。

你必须知道：

- 旧式链帮助你理解“链的类型”
- 新式链更符合现代 LangChain 的使用方向

---

## 8. 基础链：LLMChain

`LLMChain` 是历史上最经典的链之一。

## 8.1 它是什么

本质上就是：

`Prompt + LLM`

用于执行最基础的一次模型任务。

适合：

- 单次问答
- 摘要
- 分类
- 翻译

## 8.2 它的局限

课件里明确指出：

`LLMChain` 已在较新版本中被废弃趋势替代，推荐使用 `prompt | llm`

这说明什么？

说明框架设计正在从“专门的链类”走向“统一的 Runnable 组合”。

## 8.3 学 LLMChain 的意义还在吗

有。

因为它能帮助你理解：

- 链最小单位长什么样
- Prompt 和模型怎样形成一个完整任务
- 旧代码里为什么经常看到它

但在新项目里，你更应该优先掌握 LCEL 写法。

## 8.4 LLMChain 的关键参数怎么理解

课件列了一些参数，我帮你提炼出最值得理解的几个。

### `llm`

要调用的模型。

### `prompt`

要用的提示模板。

### `verbose`

是否打印详细执行日志。

这个在学习和调试阶段非常有用，因为你能看见：

- 格式化后的 prompt
- 链执行过程

### `memory`

可选记忆对象。

说明链本身也可以和上下文管理结合。

### `output_parser`

输出解析器。

这说明哪怕是最基础的链，也已经在体现 Model I/O 的完整思想。

---

## 9. SimpleSequentialChain：最简单的顺序链

这是本章最值得理解的传统链之一。

## 9.1 它解决的问题

当一个任务需要多个步骤依次执行时，就需要顺序链。

比如：

1. 先详细解释一个概念
2. 再把这个长解释压缩成一句短总结

这就是典型的串行处理流程。

## 9.2 它的特点

`SimpleSequentialChain` 的特点非常鲜明：

- 多个链按顺序执行
- 每一步都只有单一输入和单一输出
- 前一步输出自动作为下一步输入

它特别适合：

- 流程简单
- 数据传递线性
- 不需要多个变量来回映射

## 9.3 为什么它“简单”

因为它默认：

- 输入键是 `input`
- 输出键是 `output`

你不需要手动指定变量映射。

这使得它很好上手，但也带来了限制。

## 9.4 它的限制

一旦你的流程涉及：

- 多个输入变量
- 多个中间输出
- 某步需要使用更早之前的变量

那 `SimpleSequentialChain` 就不够用了。

这时候必须升级到更通用的 `SequentialChain`。

---

## 10. SequentialChain：更通用的顺序链

这是传统链体系里最重要的一个“进阶版”。

## 10.1 它和 SimpleSequentialChain 的本质区别

课件给出的核心区别是：

- `SimpleSequentialChain`：单输入 / 单输出
- `SequentialChain`：多输入 / 多输出

更准确地说：

`SequentialChain` 允许你显式定义各子链之间变量如何传递。`

## 10.2 它为什么更强

因为真实项目里，流程经常不是简单的一根线，而是：

- 一步产出多个字段
- 下一步只用其中某个字段
- 某一步还要继续用最初输入

这就需要精确的变量命名和映射。

## 10.3 它的关键设计思想

你要明确指定：

- `input_variables`
- `output_variables`
- 每个子链的 `output_key`

这本质上是在做：

`数据流显式建模`

### 一个非常重要的工程认知

链不仅是“步骤串起来”，更是“数据在步骤间怎么流动”。

`SequentialChain` 正是在解决这个问题。

## 10.4 什么时候必须用 SequentialChain

课件的例子非常典型：

### 例子1：解释 -> 总结

第一条链输出长解释，第二条链输入这段解释做摘要。

### 例子2：翻译 -> 总结 -> 识别语言 -> 按识别语言评论

这里已经明显是多变量流转。

### 例子3：查询价格 -> 生成促销文案

如果第二条链需要同时使用：

- 商品名
- 价格

那就不是简单单输入单输出了。

所以不能用 `SimpleSequentialChain`，必须用 `SequentialChain`。

### 结论

当你需要处理多变量或异构数据时，`SequentialChain` 的灵活性是必不可少的。

---

## 11. 如何理解顺序链的“变量传递”

这部分是初学者最容易卡住的地方。

## 11.1 不只是“步骤顺序”，更是“字段映射”

在简单场景里，你只感受到：

- 第一步输出给第二步

但在复杂场景里，其实你必须考虑：

- 这个输出变量叫什么名字
- 下一个链的 prompt 需要哪个变量名
- 最后我要保留哪些中间结果

所以链设计的关键，不只是顺序，而是：

`中间变量命名是否清晰`

## 11.2 为什么 output_key 很重要

因为它定义了这个子链产出的结果，在后续流程中叫什么。

如果命名混乱，就会出现：

- 数据接不上
- prompt 变量缺失
- 很难调试

### 一个实践建议

中间结果变量名尽量语义化，比如：

- `Chinese_Review`
- `Chinese_Summary`
- `Language`
- `Comment`

这种命名远比 `result1`, `result2` 好得多。

---

## 12. 顺序链背后的工程思想

这部分值得单独拎出来。

顺序链最重要的价值，不是“能多调几次模型”，而是：

### 12.1 把复杂任务拆成多个可控小任务

例如：

- 先翻译
- 再摘要
- 再识别语言
- 再评论

这比一口气让模型做所有事情，往往更可控。

### 12.2 提高可调试性

如果结果不对，你更容易定位问题出在哪一步。

### 12.3 让提示词设计更单一、更清晰

每个 prompt 只负责一个子任务，通常比“大一统神 prompt”更稳定。

### 12.4 便于替换单个环节

你可以只调整某一环：

- 改第 2 步的 prompt
- 换第 3 步的模型
- 增加第 4 步的 parser

这就是模块化工作流的价值。

---

## 13. LLMMathChain：把自然语言问题转成数学求解流程

这部分课件标注为“了解”，但它很能说明 Chain 的思想。

`LLMMathChain` 的思路不是让模型纯靠语言去心算，而是：

1. 把自然语言问题转成数学表达式
2. 再用 Python 或计算工具求值
3. 最后把结果返回

### 它揭示了什么

这说明 Chain 的价值之一是：

`把模型擅长的“理解和转换”与程序擅长的“精确执行”结合起来`

模型不一定要自己做完所有事，它也可以只是中间协调者。

---

## 14. RouterChain：为什么有时不能只靠顺序

顺序链是假设流程已经固定。

但真实业务中，有时流程不是固定的，而是要根据输入动态选择路径。

这就需要 `RouterChain`。

## 14.1 它解决什么问题

比如你有三条子链：

- 数学问题链
- 历史问题链
- 编程问题链

用户问题每次都可能不同。

你不可能固定总走同一条链。

这时候就需要先判断“属于哪类任务”，再路由到对应子链。

## 14.2 它的本质

`RouterChain` 本质上是在做 AI 工作流里的“分发器”。

从流程图角度看：

- `SequentialChain` 更像直线流水线
- `RouterChain` 更像岔路口

### 一个很好的理解方式

顺序链解决：

- 先做什么，再做什么

路由链解决：

- 到底该走哪条链

---

## 15. StuffDocumentsChain：为什么文档处理也能被抽象成链

这部分也很有代表性。

`StuffDocumentsChain` 的核心思路是：

`把多个文档拼接到一个 prompt 里，再交给模型处理`

## 15.1 它适合什么场景

适合：

- 文档总结
- 文档对比
- 全局理解型问答

前提是：

- 文档数量不多
- 文档总长度没有大到超出上下文

## 15.2 它的优缺点

### 优点

- 简单直接
- 模型能一次看到全部内容

### 缺点

- 文档一多就容易爆上下文
- token 成本高

### 一个关键认知

这类链说明 LangChain 中“链”的对象不只是模型调用，也可以是文档处理流程。

这部分是这章最有“版本意识”的内容。

课件后半部分开始介绍：

- `create_sql_query_chain`

这种写法已经体现出现代 LangChain 的方向。

## 16.1 为什么会有这个演进

因为 LangChain 正在从：

- 大量专用链类

走向：

- 更统一的 Runnable / LCEL 体系
- 更明确的工厂函数

## 16.2 这对学习者意味着什么

你不能只背老类名。

你更应该理解：

- 每种链解决什么问题
- 背后的数据流和任务流是什么
- 新式 API 为什么更偏组合式而不是继承一堆类

## 17. create_sql_query_chain：自然语言转 SQL 的典型链

这个链非常有代表性。

## 17.1 它做了什么

把用户的自然语言问题，转成可以在数据库上执行的 SQL 查询语句。

例如：

- “一共有多少个员工？”

会被转换成：

- 某条 `SELECT COUNT(*) ...` 查询

## 17.2 它的工程价值

这说明链不只是“生成文本”，还可以生成结构化中间产物，然后连接到外部系统。

这实际上就是很多 AI 应用的核心思路：

`自然语言 -> 可执行中间表示 -> 外部系统执行 -> 返回结果`

### 一个重要理解

很多看起来“很智能”的 AI 系统，本质上就是这类多步链路。

## 18. create_stuff_documents_chain：LCEL 风格的文档链

它可以看作是旧式 `StuffDocumentsChain` 的新式版本化表达。

它体现的是：

- 任务目标更明确
- 与现代 Runnable 体系更一致

### 本质上，这类工厂函数在做什么

它不是在发明新思想，而是在把常见链路模式封装成更标准的构造方式。

这对开发者很友好，因为你不用每次都从零搭流程。

## 20. 初学者最容易踩的坑

## 20.1 误以为 Chain 只是语法糖

不是。

它本质上是工作流组织方式。

## 20.2 误以为链越长越高级

也不是。

链应该围绕任务拆分需要来设计，而不是为了显得复杂。

## 20.3 误以为 SimpleSequentialChain 能处理所有顺序场景

不能。

一旦涉及多个变量映射，必须考虑 `SequentialChain`。

## 20.4 误以为只要模型够强，就不需要拆步骤

这是很常见的误区。

很多任务拆成多步后反而：

- 更稳定
- 更容易调试
- 更容易控制

## 20.5 误以为老链都不用学

也不对。

虽然很多新项目更推荐 LCEL，但老代码、老教程和面试问题里依然常见这些概念。

## 21. 如何做链的选型

这部分非常实战。

## 21.1 如果只是一次简单任务

例如：

- 翻译
- 摘要
- 单次问答

直接用：

`prompt | model | parser`

通常就够了。

## 21.2 如果任务是单线串行，且每步只有一个输入输出

例如：

- 先解释，再总结

可以用：

- `SimpleSequentialChain`

或者直接用 LCEL 逐步组合。

## 21.3 如果任务涉及多个变量流转

例如：

- 查询价格后写营销文案
- 翻译后摘要再评论

优先考虑：

- `SequentialChain`

## 21.4 如果任务要动态分流

例如：

- 用户问题可能属于不同领域

考虑：

- `RouterChain`

## 21.5 如果输入是文档集合

考虑：

- `StuffDocumentsChain`
- `create_stuff_documents_chain`

## 21.6 如果任务本身有成熟模式

例如：

- 文本转 SQL
- 数学求解

优先看看是否已有现成链工厂。

## 22. 从这一章应该学到的核心工程方法论

这一章真正重要的，不是认识多少链类，而是学到下面这些方法论。

### 22.1 把复杂任务拆成步骤

不要指望一个 prompt 永远解决所有问题。

### 22.2 明确中间数据长什么样

链设计的核心，是中间结果如何命名、如何传递。

### 22.3 优先使用统一抽象

LCEL 和 Runnable 的意义就在于降低组合复杂度。

### 22.4 流程设计比模型堆砌更重要

很多系统的能力提升，不一定来自“换更贵模型”，而来自：

- 更好的任务拆分
- 更清晰的数据流
- 更合适的链类型

### 22.5 新旧 API 都要会辨认

理解 Legacy Chains 的概念，同时优先掌握 LCEL 的现代写法。

## 23. 这章最值得你记住的几句话

### 1. Chain 是可复用 AI 工作流，不是单次模型调用

### 2. LCEL 是现代 LangChain 里构建链的核心方式

### 3. Runnable 统一了不同组件的调用协议

### 4. SimpleSequentialChain 适合单输入单输出的线性流程

### 5. SequentialChain 适合多变量、多中间结果的复杂流程

### 6. RouterChain 解决“走哪条链”的问题

### 7. 现成链工厂体现了 LangChain 对常见业务模式的抽象

## 24. 最后总结

这一章《Chains》本质上是在帮你完成一个认知升级：

从：

- “我会调用大模型”

升级到：

- “我会设计由多个 AI 步骤组成的工作流”

如果把它压缩成最核心的三层理解，就是：

### 第一层：链是什么

链就是把多个组件串起来形成流程。

### 第二层：为什么需要链

因为真实任务往往不止一步，而且中间结果需要管理和复用。

### 第三层：怎么选链

- 简单任务：LCEL 最小链
- 简单串行：SimpleSequentialChain
- 多变量流转：SequentialChain
- 动态分发：RouterChain
- 文档处理：StuffDocumentsChain / create_stuff_documents_chain
- 专项任务：create_sql_query_chain 等

如果你已经懂计算机基础，那么学完这章后，最值得你立刻去做的练习是：

1. 自己写一个 `prompt | model | parser` 的最小链
2. 自己写一个“解释 -> 总结”的顺序链
3. 自己写一个“翻译 -> 总结 -> 评论”的多变量顺序链

把这三件事练熟，你对 LangChain 工作流的理解会明显上一个台阶。
