# 《05-LangChain使用之Tools》学习笔记版

## 1. 这一章在讲什么

这一章的主题是 `Tools`。

如果前面的 `Model I/O` 解决的是“怎么让大模型说话”，那么这一章解决的是：

- 怎么让大模型“干活”
- 怎么让大模型接入外部能力
- 怎么让 AI 不只是生成文本，而是能够操作真实世界中的资源

最核心的一句话是：

`Tools = 给大模型提供可调用的外部能力`

这些能力可以是：

- 搜索
- 计算
- 查询数据库
- 调用 API
- 操作文件
- 执行代码

所以，`Tools` 不是让模型变聪明，而是让模型变得“有手有脚”。

---

## 2. 为什么需要 Tools

很多 AI 初学者会有一个误区：觉得大模型什么都会。

其实不对。

大模型真正擅长的是：

- 语言理解
- 语言生成
- 模式归纳
- 根据上下文做推断

但它天然不擅长或者根本做不到：

- 获取实时信息
- 精确计算复杂表达式
- 访问你的本地文件
- 操作数据库
- 调用企业内部系统
- 真正执行某个动作

比如用户问：

- “帮我查一下今天上海天气”
- “把这个文件移动到桌面”
- “帮我查询订单号 12345 的物流状态”

如果没有工具，大模型最多只能“描述它可能怎么做”，但不能真的做。

而 `Tools` 的意义就在这里：

- 让模型负责“理解需求和决策”
- 让工具负责“真正执行动作”

你可以把它理解成一种分工：

- `LLM` 负责大脑
- `Tool` 负责执行器

这也是 LangChain 从“提示工程框架”走向“AI 应用框架”的关键一步。

---

## 3. 用一句人话理解 Tool

Tool 本质上就是一个“给模型看的函数接口”。

注意，这里有两个层面：

### 3.1 对程序员来说

它通常对应一个函数、类方法，或者一个被封装好的能力模块。

### 3.2 对大模型来说

它是一段可供选择的“能力说明书”，里面会告诉模型：

- 这个工具叫什么
- 它能做什么
- 需要什么参数
- 参数是什么类型

模型看到这些信息后，就能在合适的时候说：

- “这个问题我应该调用 `search` 工具”
- “这个请求我应该调用 `move_file` 工具”

所以从本质上说，Tool 是连接两端的桥梁：

- 一端是自然语言世界
- 一端是程序执行世界

---

## 4. Tools 的核心价值

根据课件内容和工程实践，Tools 的价值可以概括为 3 点。

### 4.1 增强 LLM 的功能边界

大模型原本只能输出文本。

接入 Tools 后，它就能：

- 搜索信息
- 访问外部服务
- 读写文件
- 查询数据库
- 触发业务逻辑

也就是说，模型从“会说”变成了“能做”。

### 4.2 支持智能决策

在 Agent 场景中，模型不是固定调用某个工具，而是会先判断：

- 当前问题需不需要工具
- 如果需要，应该调用哪个工具
- 工具需要哪些参数

这就是“智能决策”的来源。

因此，Tool 不只是一个函数，更是 Agent 决策空间中的一个“可选动作”。

### 4.3 模块化设计，便于组合

一个 Tool 通常只做一件事。

例如：

- 搜索工具只负责搜索
- 计算工具只负责计算
- 天气工具只负责查天气
- 文件工具只负责文件操作

这种设计带来的好处是：

- 复用性高
- 易于测试
- 易于组合
- 易于扩展

复杂 AI 应用往往不是一个超大函数，而是一组小工具的协同。

## 5. Tool 的本质结构

课件里提到，Tool 一般包含几个核心要素。

这是整章最重要的知识点之一。

### 5.1 `name`

工具名称。

它是模型识别这个工具的“名字”，在一个工具列表里通常要求唯一。

比如：

- `search`
- `move_file`
- `get_weather`
- `query_order`

为什么它重要？

因为模型在选择工具时，首先看到的就是名字。

如果名称模糊不清，比如叫：

- `tool1`
- `handler`
- `process_data`

模型就很难准确理解它的用途。

所以好的工具名应该：

- 简洁
- 明确
- 有动作含义

### 5.2 `description`

工具描述。

这是给模型看的“说明文档”。

模型会根据描述判断：

- 什么时候该用它
- 它适合解决什么问题
- 它和其他工具有什么区别

比如：

- “用于回答当前事件相关问题”
- “用于根据订单号查询物流状态”
- “用于把文件从一个路径移动到另一个路径”

描述写得越清晰，模型选错工具的概率越低。

这是初学者特别容易忽略的点：

很多人只写了函数，却没有认真写 `description`，结果模型根本不会用，或者乱用。

### 5.3 `args_schema`

参数模式，一般用 Pydantic 模型描述。

它的作用是定义：

- 工具需要哪些参数
- 每个参数是什么类型
- 参数的语义说明是什么

例如一个加法工具可能要求：

- `a: int`
- `b: int`

一个搜索工具可能要求：

- `query: str`

如果再加上 `Field(description="...")`，模型就会更容易理解参数含义。

比如：

- `query: str = Field(description="要检索的关键词")`

这比只有一个裸参数 `query` 更清楚。

### 5.4 实际执行函数

Tool 不是只有元信息，它最终还要绑定一个真实函数。

比如：

```python
def add_number(a: int, b: int) -> int:
    return a + b
```

或者：

```python
def search_function(query: str):
    return "LangChain"
```

模型负责“决定调用”，程序负责“真正执行”。

### 5.5 `return_direct`

这个属性和 Agent 场景关系很大。

它表示：

- 工具执行完以后，结果是否直接返回给用户

如果 `return_direct=False`：

- 工具执行结果会先交回 Agent
- Agent 还可以继续推理、继续决定下一步

如果 `return_direct=True`：

- 工具结果直接作为最终结果返回
- 当前 Agent 流程到这里就结束

你可以把它理解为：

- `False` 更像“中间步骤”
- `True` 更像“最终答案”

## 6. Tool 是怎么工作的

课件里给出了一个非常重要的三步流程。

### 6.1 第一步：把工具信息提供给 LLM

程序先把这些信息告诉模型：

- 工具名
- 工具描述
- 参数结构

这一步的本质，是把工具转成模型能理解的上下文。

### 6.2 第二步：模型决定是否调用工具

模型根据用户输入做判断：

- 要不要调用工具
- 调哪个工具
- 参数是什么

这里要特别注意：

模型此时通常只是“返回一个调用意图”，并不一定真的执行了工具。

这是初学者最容易搞混的地方。

模型说“我要调 `move_file`”，不等于文件已经真的移动了。

### 6.3 第三步：程序根据模型返回结果，真正执行工具

你的代码需要：

- 读取模型返回的工具调用信息
- 解析工具名和参数
- 找到对应工具
- 真正执行
- 再决定是否把结果返回给用户，或者继续后续流程

所以完整流程不是：

`用户 -> 模型 -> 结束`

而是：

`用户 -> 模型判断 -> 程序执行工具 -> 返回结果`

这就是为什么工具调用本质上是“模型决策 + 程序落地”的联合机制。

---

## 7. 为什么工具名称、描述、参数模式必须认真设计

课件里专门强调了一点：

如果工具的 `name`、`description` 和 `JSON Schema` 设计得更好，模型表现会更好。

这背后的原因非常工程化。

模型并不是“直接看懂你的 Python 函数”，它主要依赖你暴露给它的接口信息来判断怎么用。

因此，工具接口设计其实和 API 设计非常像。

如果接口设计差，会出现这些问题：

- 模型不知道什么时候该调用它
- 模型把工具用错场景
- 模型参数传错
- 多个工具功能重叠，模型不知道选哪个

一个高质量 Tool 通常满足：

- 名称有辨识度
- 描述清楚边界
- 参数名有业务语义
- 参数说明足够明确

比如下面两个描述相比：

差的描述：

- `处理数据`

好的描述：

- `根据用户输入的城市名称查询当前天气信息`

显然第二种更容易让模型做对决策。

## 8. 自定义 Tool 的两种方式

课件中讲了两种主流方式。

---

## 9. 方式一：`@tool` 装饰器

这是最简单、最常用、也最适合初学者入门的方式。

基本写法：

```python
from langchain.tools import tool

@tool
def add_number(a: int, b: int) -> int:
    """两个整数相加"""
    return a + b
```

这个写法做了几件事：

- 把普通函数包装成 LangChain Tool
- 默认使用函数名作为工具名
- 默认使用函数文档字符串作为工具描述
- 自动从函数签名推断参数结构

### 9.1 初学者必须注意的点

#### 1. 要写文档字符串

课件里特别提到，函数必须提供文档字符串。

因为默认情况下，Tool 的 `description` 会来自函数的 docstring。

如果你不写，工具的可读性和可用性都会下降。

#### 2. 函数签名会影响参数结构

例如：

```python
def add_number(a: int, b: int) -> int:
```

LangChain 会推断出参数：

- `a` 是整数
- `b` 是整数

这就构成了给模型看的参数模式。

#### 3. 调用方式不是直接传位置参数

课件例子里使用的是：

```python
res = add_number.invoke({"a": 10, "b": 20})
```

这说明 Tool 在 LangChain 中通常通过统一接口调用，而不是像普通函数那样直接 `add_number(10, 20)`。

这样设计的好处是：

- 统一框架接口
- 更方便接入 Agent
- 更方便做参数校验

### 9.2 `@tool` 方式的优点

- 写法简单
- 上手快
- 代码量少
- 适合简单工具封装

### 9.3 适合什么场景

- 快速做 demo
- 自定义一些轻量函数型工具
- 学习 LangChain Tool 的基本机制

---

## 10. `@tool` 的可配置参数

课件还展示了可以在装饰器中直接覆盖默认行为。

例如：

```python
@tool(
    name_or_callable="add_two_number",
    description="two number add",
    return_direct=True
)
def add_number(a: int, b: int) -> int:
    """两个整数相加"""
    return a + b
```

这说明你可以手动指定：

- 工具名
- 工具描述
- 是否直接返回

这在工程里很常见。

因为函数名可能更偏向程序内部命名，而工具名更偏向给模型理解。

比如你的函数名可能叫：

- `calc_sum_internal`

但给模型的工具名更适合叫：

- `add_numbers`

---

## 11. 用 `args_schema` 增强参数表达力

课件展示了配合 Pydantic `BaseModel` 和 `Field` 的做法：

```python
from pydantic import BaseModel, Field

class FieldInfo(BaseModel):
    a: int = Field(description="第1个参数")
    b: int = Field(description="第2个参数")
```

再绑定到工具：

```python
@tool(
    name_or_callable="add_two_number",
    description="two number add",
    args_schema=FieldInfo,
    return_direct=True
)
def add_number(a: int, b: int) -> int:
    """两个整数相加"""
    return a + b
```

这样做的意义非常大。

### 11.1 它不仅仅是“更规范”

它实际上提升了三件事：

- 模型理解能力
- 参数校验能力
- 接口可维护性

### 11.2 为什么对初学者尤其重要

你现在可能觉得：

- “参数不就两个 int 吗，有必要吗？”

但一旦工具稍微复杂一点，比如：

- 查询酒店：城市、入住日期、离店日期、房型、人数
- 查询订单：订单号、用户 ID、是否包含物流详情

如果没有结构化参数说明，模型就很容易传错。

所以从工程角度说：

- 简单工具可以靠函数签名推断
- 稍复杂工具最好显式定义 `args_schema`

---

## 12. 方式二：`StructuredTool.from_function`

第二种自定义方式是：

```python
StructuredTool.from_function(...)
```

课件中强调，这种方式比 `@tool` 更灵活、可配置性更强。

示例逻辑大致是：

```python
from langchain_core.tools import StructuredTool

def search_function(query: str):
    return "LangChain"

search1 = StructuredTool.from_function(
    func=search_function,
    name="Search",
    description="useful for when you need to answer questions about current events"
)
```

### 12.1 它和 `@tool` 的关系

你可以把它理解为：

- `@tool` 是更简洁的语法糖
- `StructuredTool.from_function` 是更显式的构造方式

### 12.2 它的优势

- 配置更明确
- 代码可读性更强
- 更适合复杂工具定义
- 更适合同步/异步等扩展场景

### 12.3 什么时候更推荐用它

当你遇到这些情况时，往往更适合：

- 工具配置较多
- 需要更明确地定义 schema
- 希望工具构造逻辑更显式
- 项目代码追求统一工厂式写法

---

## 13. `@tool` 和 `StructuredTool.from_function` 怎么选

可以这样理解：

### 13.1 如果你是初学者

优先用 `@tool`。

原因：

- 简单
- 好理解
- 学习成本低

### 13.2 如果你在写正式项目

根据复杂度决定。

通常：

- 简单工具：`@tool`
- 复杂工具或需要更细配置：`StructuredTool.from_function`

### 13.3 本质区别

它们都能把函数包装成 Tool。

区别主要不在“能不能做”，而在：

- 写法风格
- 显式程度
- 配置弹性

## 14. 工具调用不是“自动魔法”，而是一个明确流程

这一部分是整章最值得你真正理解的地方。

很多新手会误以为：

- “我把 tools 传给模型，模型就自己把文件移动了”

其实不是。

课件里的例子说明了真实过程。

---

## 15. 工具调用示例的核心思想

课件使用了 `MoveFileTool` 做演示，并把工具转换成 OpenAI 可识别的函数定义，然后传给模型。

当用户输入：

- “将文件 a 移动到桌面”

模型看到：

- 有一个叫 `move_file` 的工具
- 它需要路径参数
- 用户需求和这个工具高度匹配

于是模型返回的不是普通自然语言，而是一个“函数调用意图”。

表现形式类似：

```python
AIMessage(
    content="",
    additional_kwargs={
        "function_call": {
            "name": "move_file",
            "arguments": "{\"source_path\":\"a\", \"destination_path\":\"...\"}"
        }
    }
)
```

这说明什么？

说明模型在说：

- “我建议调用 `move_file`”
- “参数我也帮你组织好了”

但是这一步仍然只是“建议调用”。

真正执行文件移动的，还是你的代码。

---

## 16. 模型为什么有时调用工具，有时不调用

课件里还专门做了对照实验。

如果用户问的是：

- “今天的天气怎么样？”

但你只给了模型一个 `MoveFileTool`，那么模型通常不会瞎调这个工具。

它会直接返回普通文本回复。

这说明模型在做一层匹配判断：

- 用户需求是否和某个工具相关

如果相关：

- 返回 `function_call`

如果不相关：

- 返回自然语言内容

这体现出工具调用并不是机械触发，而是“基于语义匹配的决策行为”。

---

## 17. 如何判断模型是否请求了工具调用

课件中的代码逻辑是：

```python
if "function_call" in response.additional_kwargs:
    ...
else:
    ...
```

这一步非常关键。

也就是说，在程序层面你不能假设“每次都会调用工具”，你必须检查模型返回值。

程序一般要做两条分支处理：

### 17.1 分支一：模型请求工具调用

需要做：

- 取出工具名
- 取出参数字符串
- 反序列化 JSON
- 找到对应工具
- 执行工具

### 17.2 分支二：模型没有调用工具

直接把自然语言结果返回给用户即可。

这其实就是一个典型的“路由逻辑”。

---

## 18. 真正执行工具的过程

课件中的逻辑是：

1. 从 `response.additional_kwargs["function_call"]` 中提取：
   - `name`
   - `arguments`
2. 把 `arguments` 从 JSON 字符串解析成 Python 字典
3. 根据工具名选择对应工具
4. 调用工具执行
5. 输出执行结果

这一段非常像后端开发里的“RPC 调度”或者“函数分发”：

- 模型负责发出调用请求
- 你的程序负责路由和执行

所以从软件架构角度看，LLM 更像“控制器”或“决策层”，工具才是“能力实现层”。

---

## 19. 这一章和 Agent 的关系

课件里有一句很关键的话：

- 后续工具的调用动作可以由 Agent 接管

这说明：

- 这一章先教你理解 Tool 本身
- 下一步才是让 Agent 自动接管“判断、调用、再思考”的循环

换句话说：

- `Tools` 是能力单元
- `Agent` 是工具编排者

没有 Tools，Agent 没有可执行动作。
没有 Agent，Tools 往往需要你手动接线。

所以你可以把学习顺序理解为：

1. 先学会定义能力
2. 再学会让智能体动态使用能力

## 20. 从工程实践看，这一章最应该真正吃透什么

如果你不是只想“看懂代码”，而是想真正理解 LangChain，这一章至少要吃透下面 6 个点。

### 20.1 Tool 不等于普通函数

普通函数只对程序员可见。

Tool 是：

- 被包装过的函数
- 带元信息的能力接口
- 面向模型暴露的可调用能力

### 20.2 Tool 的设计质量会直接影响模型效果

模型不是读你的心，它主要依赖：

- 名称
- 描述
- 参数 schema

所以 Tool 设计本身就是提示工程的一部分。

### 20.3 模型只是决定要不要调，不一定真的执行

这一点特别重要。

模型返回的 `function_call` 本质上是一个结构化意图，不是副作用本身。

### 20.4 程序仍然掌握最终执行权

这很重要，也很安全。

因为你完全可以在真正执行前加上：

- 参数检查
- 权限检查
- 路径白名单
- 风险拦截
- 人工确认

这也是为什么 Tool 机制适合真正落地业务系统。

### 20.5 `return_direct` 决定结果回路

它影响的是 Agent 在工具执行后是否继续推理。

这个选项本质上影响的是“控制流”。

### 20.6 Tool 是 Agent 的基础设施

你后面学习 Agent 时会发现：

- Agent 的核心能力不是“更聪明”
- 而是“会在恰当的时候选对工具”

所以 Tool 是 Agent 能力的地基。

## 22. 一段更专业的总结

从专业角度看，LangChain Tools 的本质是：

- 将外部能力以结构化接口形式暴露给 LLM
- 让 LLM 基于自然语言语义做工具选择和参数生成
- 再由宿主程序完成真实执行和结果回传

因此，Tools 解决的不是“模型会不会说”，而是：

- 模型如何和外部系统协作
- AI 应用如何从生成式能力走向执行式能力

这一步是构建 AI Agent、业务 Copilot、自动化助手、企业智能应用的关键基础设施。

## 23. 本章重点速记

- `Tools` 用于扩展 LLM 的外部能力，让模型不只是生成文本，还能调用外部系统、API 和函数。
- 一个 Tool 通常由 `name`、`description`、`args_schema`、执行函数、`return_direct` 等部分组成。
- `name`、`description`、参数说明写得越清楚，模型越容易选对工具、传对参数。
- 自定义工具有两种常见方式：`@tool` 和 `StructuredTool.from_function`。
- `@tool` 更适合入门和轻量场景，`StructuredTool.from_function` 更适合配置更复杂的场景。
- 模型返回工具调用请求，不代表工具已经执行；真正执行动作的是程序代码。
- 判断是否调用工具，通常要检查 `response.additional_kwargs` 中是否存在 `function_call`。
- `return_direct=True` 表示工具结果可直接作为最终结果返回；`False` 表示结果还会继续交给 Agent 处理。
- Tool 是 Agent 的基础，没有 Tool，Agent 很难真正与外部世界交互。

## 25. 版本补充说明

这份课件中的工具调用示例采用的是课程当时的函数调用写法，例如 `functions=...` 和 `function_call` 风格的返回结构。

在 LangChain 和 OpenAI 的新版本生态里，工具调用接口可能会进一步演化，例如出现更统一的 `tool calling` 风格 API。

但你不用焦虑。

因为无论 API 名字怎么变，这一章真正不变的底层思想始终是：

- 把工具能力描述给模型
- 模型决定是否调用和如何传参
- 程序负责真实执行

只要你把这个底层机制理解透了，后续适配新版本并不难。
